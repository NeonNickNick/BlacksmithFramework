# 项目架构

本文档面向以下读者：

- 想继续维护 Blacksmith 的开发者
- 想定位规则、回合结算或前端展示问题的调试者
- 想基于当前引擎编写大型 Mod、职业包或 AI 的扩展开发者

目标是把下面几件事讲清楚：

- 程序是如何启动并完成装配的
- 一局游戏在内存中由哪些核心对象组成
- 一个技能从"声明"到"真正生效"经历了哪些阶段
- Web 宿主、前端快照、AI、Mod 分别插在哪一层
- 当你需要扩展或排障时，应该先看哪里

---

## 1. 总体设计思路

当前仓库可以概括为"基础设施 + 单宿主 + 核心库 + 示例扩展"四层结构：

- **`ClapInfra`** — 最底层的基础设施库。提供 `ClapEnum<T>` 可扩展枚举框架、`ClapSkillPackage` 技能包反射配对机制、`ClapBody` 实体组件模板、`ClapTurnContext` 通用决议上下文、`ClapJudger`/`ClapJudgeRuleManager`/`ClapIntent` 判定管线骨架、`IClapDSLSourceFile` DSL 到 Intent 编译契约，以及 `DllLoader` 插件扫描工具。不依赖任何其他项目，是纯抽象的框架级代码。
- **`BlacksmithClient`** — 运行入口。负责加载插件、初始化 AI、启动本地 Web 宿主、暴露最小 API、托管静态前端。
- **`BlacksmithCore`** — 真正的游戏引擎。包括领域模型、技能系统、DSL、判定引擎、动态规则、AI 接口、插件加载、职业注册、Web 会话封装。它的可扩展枚举和技能包系统直接建立在 ClapInfra 之上。
- **`ModExamples`** — 示例扩展目录。用于演示"扩展资源/防御枚举 + 新职业 + Common 修改器"的写法。

和传统"前后端分离"的项目不同，这个仓库当前不是一个远程服务端加独立前端工程，而是：

1. 由 `BlacksmithClient` 启动 ASP.NET Core 本地站点
2. 用 `wwwroot` 托管一个轻量静态页面
3. 用几个 `/api/*` 端点操作同一进程里的 `GameInstance`

也就是说，前端、会话层、规则引擎默认跑在同一个进程里。规则对象不需要额外序列化成复杂服务协议，但 Web 层与引擎层的边界要靠"快照对象"和"会话门面"人为保持清晰。

## 3. ClapInfra：基础设施层

ClapInfra 是整个框架的底层抽象，不包含任何游戏特定逻辑。它的设计目标是让 BlacksmithCore 中的可扩展枚举、技能包和实体组件都建立在统一的基础上。

### 3.1 `ClapEnum<T, TMemberAttribute>` — 可扩展枚举框架

传统的 C# `enum` 编译后无法追加成员。`ClapEnum` 通过"运行时可创建成员 + 稳定可比较值"的方式解决了这个问题。

核心类型层次：

```
IClapEnum                    → 接口：GetBEValueType(), Create(name, priority)
  ↑
ClapEnum                     → 抽象基类：持有静态 _isOpen 开关，提供 CloseFactory()
  ↑
ClapEnum<T, TMemberAttribute> → 泛型模板：单例模式，反射扫描带特性的方法自动注册成员
  ↑
BlacksmithEnum<T>             → Blacksmith 的游戏用封装（BlacksmithCore 中定义）
```

`BEValue` 是一个嵌套结构体，内部包含：
- `_uniqueID`：全局递增的唯一标识，用于相等性比较
- `_priority`：排序优先级，用于 `CompareTo` 和防御/攻击/阶段排序

关键行为：
- 构造函数通过反射扫描所有带 `TMemberAttribute` 特性的 `public` 实例方法（无参，返回 `BEValue`），自动调用 `Create(methodName, priority)` 注册成员。
- `Create(name, priority)` 直接覆盖同名成员——这是 Mod 修改现有枚举优先级的基础。
- 启动完成后调用 `ClapEnum.CloseFactory()`，此后任何 `new BEValue(...)` 都会抛出异常。

### 3.2 `ClapSkillPackage` — 技能包反射配对

技能包系统的核心约定是"每个技能由一对方法定义"：

- `XxxCheck(ISkillContext)` → `bool`：技能是否可用
- `Xxx(ISkillContext)` → `IDSLSourceFile`：技能的具体实现

`ClapSkillPackage` 在构造时反射扫描当前类的所有 `private` 实例方法：

1. 找到以 `Check` 结尾、返回 `bool`、参数为 `TISkillContext` 的方法
2. 去掉 `Check` 得到技能名（转小写）
3. 在同类型中查找同名、返回 `TIDSLSourceFile`、参数为 `TISkillContext` 的方法
4. 配对成功则创建委托存入字典

其中 `TISkillContext` 和 `TIDSLSourceFile` 是泛型参数。Blacksmith 中分别指定为 `ISkillContext` 和 `IDSLSourceFile`。**注意**：技能方法必须返回接口 `IDSLSourceFile`，返回具体类 `DSL.SourceFile` 会导致反射配对失败、技能静默不可用。

`PackageType` 枚举区分两类技能包：
- `Main`：主职业包，构造时会调用 `AddModOnInit()` 来合并修改器
- `Modifier`：修改器包，挂到目标职业上提供额外技能

### 3.3 `ClapBody` — 实体组件模板

`ClapBody` 是一个基于类型的组件容器。构造函数接收 `HashSet<object>`，按每个对象的 `GetType()` 存储到内部字典：

```csharp
_components[obj.GetType()] = obj;
```

通过 `Get<T>()` 方法按类型取出组件。`Update()` 方法遍历所有实现了 `IUpdatePerRound` 的组件并调用其 `Update()`。

Blacksmith 的 `Body` 继承 `ClapBody`，在构造时注入七个组件：`Community`、`Skill`、`Health`、`Defense`、`Resource`、`Effect`、`TurnContext`。

### 3.4 `ClapTurnContext` — 通用决议上下文

`ClapTurnContext<TIResolution, TCommunity>` 是一个泛型抽象的回合上下文。构造时接收一个 `HashSet<Type>`，为每种决议类型创建对应的 `List<T>`：

```csharp
_resolutionLists[type] = new List<T>();
```

提供三个核心方法：
- `WriteResolution(resolution)`：按类型写入决议
- `Get<TResolution>()`：按类型取出决议列表
- `Execute<TResolution>(community)`：执行指定类型的所有决议

Blacksmith 的 `TurnContext` 注册了四种决议类型：`AttackResolution`、`DefenseResolution`、`ResourceResolution`、`EffectResolution`。其 `ExecuteImpl` 实现了延迟回合机制——只执行 `DelayRounds == 0` 的决议，并对未执行的决议递减 `DelayRounds`。

### 3.5 `IClapDSLSourceFile` — DSL 到 Intent 编译契约

`IClapDSLSourceFile<TCommunity, TJudger, TJudgeRuleManager, TIntent, TIDSLSourceFile>` 是一个 CRTP（奇异递归模板模式）风格的泛型接口，定义了 DSL 源文件到判定意图的编译契约：

- `Compile(TJudger? judger)` → `TIntent`：将 DSL 编译为意图，可选传入 Judger 以注册动态规则。

Blacksmith 中通过 `IDSLSourceFile` 接口绑定所有泛型参数：
```csharp
public interface IDSLSourceFile : IClapDSLSourceFile<Community, Judger, JudgeRuleManager, Intent, IDSLSourceFile> { }
```

这个接口使得判定循环（`ClapJudger`）只需知道"能编译成 Intent 的东西"，而不依赖 `SourceFile` 的内部结构。

### 3.6 `ClapJudger` — 判定主循环骨架

`ClapJudger<TCommunity, TJudger, TJudgeRuleManager, TIntent, TIDSLSourceFile>` 提供回合判定的主循环骨架：

- 持有 `Player` 和 `Enemy`（`TCommunity`）
- 持有 `JudgeRuleManager`（通过 `new()` 自动创建）
- `Judge(playerSfs, enemySfs)` 编排完整流程：Compile → TranslateIntentsToResolutions → GetRule 执行

Blacksmith 的 `Judger` 继承 `ClapJudger`，只重写 `Compile` 处理被动技能插入顺序（被动技能 Intent 插在最前面），并添加 `Swap()` 方法用于视角交换。

### 3.7 `ClapJudgeRuleManager` — 规则管理器抽象

`ClapJudgeRuleManager<TCommunity>` 是规则管理器的抽象基类，只定义一个抽象方法：

- `GetRule()` → `Action<TCommunity, TCommunity>`：返回本回合的复合判定规则。

Blacksmith 的 `JudgeRuleManager` 继承它，实现了 11 阶段的 `StageRuleContainer` + `RuleUnit`（base/override/before/after）完整规则引擎。

### 3.8 `ClapIntent` — 意图抽象包装

`ClapIntent<TCommunity>` 是判定意图的抽象基类，包含一个 `Action<TCommunity> Execute` 属性。Blacksmith 的 `Intent` 继承它（空类体），具体行为通过 DSL 编译时组合的 `Action` 闭包实现。

### 3.9 `DllLoader` — 程序集扫描工具

`DllLoader` 提供通用的程序集加载和类型发现能力：

- `Initialize(folderPath)`：加载目录下所有 `*.dll`
- `LoadByType<T>()`：扫描所有已加载程序集，实例化所有 `T` 的实现类
- `LoadStaticByAttribute(attributeType, process)`：扫描带指定特性的静态类并回调处理

Blacksmith 的 `PluginLoader` 持有 `DllLoader` 实例，调用其扫描能力后，将结果注册到 Blacksmith 特定的 `BlacksmithEnumRegistry` 和 `ProfessionRegistry`。

## 4. 启动链路

启动入口：`Project/Blacksmith/BlacksmithClient/Program.cs`

```csharp
PluginLoader.Initialize(AppContext.BaseDirectory);
// 创建 AI 策略列表
LocalHost.Start(strategies);
```

### 4.1 `PluginLoader.Initialize`

这一步的内部顺序非常关键：

**第一步：扫描 DLL**
遍历运行目录全部 `*.dll`，用 `Assembly.LoadFrom` 加载并缓存。

**第二步：加载 BlacksmithEnum**
1. 扫描所有 `IBlacksmithEnum` 的实现类并实例化
2. 通过 `BlacksmithEnumRegistry.RegistBlacksmithEnum` 注册
3. 扫描带 `[IsBlacksmithEnumModifier]` 的静态类
4. 读取其中带 `[IsBlacksmithEnumMember]` 的扩展方法（签名：`static BEValue Method(this EnumType e)`）
5. 调用 `targetEnum.Create(methodName, priority)` 追加或覆盖枚举成员
6. 调用 `ClapEnum.CloseFactory()` 关闭枚举工厂

**第三步：加载职业**
1. 扫描所有 `SkillPackageBase` 子类并实例化
2. 第一轮：注册所有 `PackageType == Main` 的职业名到 `ProfessionRegistry.Professions`
3. 第二轮：注册所有 `PackageType == Modifier` 的修改器（通过 `[IsProfessionModifier(targetName)]` 特性确认目标）

这里有一个重要的设计细节：`SkillPackageBase` 构造时，如果 `PackageType == Main`，会调用 `ProfessionRegistry.AddModOnInit(this)`。这个方法会查找所有目标为该职业名的修改器，将修改器中的技能（`AvailableSkillNames`、`SkillChecker`、`SkillSourceFileGenerator`）复制到主职业包上。这意味着修改器的技能在启动时就已装配完成，运行时无需再查询修改器。

### 4.2 AI 初始化

当前宿主默认创建两种 AI：
- `BloodSigilStrategy`：启发式规则 AI
- `GeneralStrategy`：MCTS 搜索 AI，可接受 `GeneralStrategyParams` 参数

### 4.3 Web 宿主启动

`LocalHost.Start(strategies)` 创建 ASP.NET Core 应用：
- 配置 `WebRootPath` 和 `ContentRootPath` 指向当前目录
- 绑定 `http://localhost:5000`
- 启用默认文件和静态文件中间件
- 创建 `WebGameSession` 实例
- 映射四个 API 端点
- 自动打开浏览器

## 5. Web 宿主与 API 层

### 5.1 `LocalHost` 的职责边界

`LocalHost` 是"HTTP 适配器"，不是"游戏控制器"。它的职责仅限于：
- 配置 ASP.NET Core 宿主
- 把 HTTP 请求翻译成 `WebGameSession` 调用
- 返回 JSON 响应

它不直接操作 `Judger`、`Body` 或任何领域对象。

### 5.2 API 一览

#### `GET /api/strategies`
返回可选模式列表。前 N 项为 AI 模式（格式：`{id: i, name: "Xxx (AI)"}`），最后一项为 `{id: N, name: "Manual"}`。

#### `POST /api/start`
输入 `{mode}`。创建新的 `GameInstance`，重置模式，若为 AI 模式则调用 `activeAI.Init(game)`。返回 `{ok, snapshot}`。

#### `POST /api/declare`
输入 `{skillName, param, esn, ep}`。

流程：
1. 校验玩家技能（`TryDeclare`）
2. 若为 AI 模式，调用 `activeAI.ChooseSkill(...)` 获取敌方技能
3. 若为手动模式，校验敌方技能（`ETryDeclare`）
4. 调用 `GameInstance.Declare(...)` 执行回合
5. 返回 `{ok, message, snapshot}`

#### `GET /api/status`
返回当前最新快照。适用于前端初始化和调试。

### 5.3 `WebGameSession`

`WebGameSession` 是 Web 层最关键的门面对象，维护：
- `_strategies`：可用 AI 列表
- `_game`：当前 `GameInstance`
- `_activeAI`：当前绑定的 AI（手动模式为 null）
- `_mode` / `_modeName` / `_isManual`

其核心职责是：
1. 管理游戏生命周期（创建、启动）
2. 校验输入（技能名合法性、游戏状态）
3. 根据模式决定敌方行动来源
4. 调用领域对象的 `GetView()` 方法，组装前端快照

快照构造直接调用 `Body.GetView()` 返回的 `BodyView`，再映射到前端所需的 JSON 形状。不再通过独立的 `WebGameReader` 类。

## 6. 运行时核心对象

### 6.1 `GameInstance`

`GameInstance` 是一局游戏的根容器，持有四个核心字段：

- `Community Player`：玩家阵营
- `Community Enemy`：敌方阵营
- `Judger Judger`：回合裁判
- `GameHistory History`：历史记录

关键方法：

- `TryDeclare(skillName, param)`：以 Player 视角检查技能合法性，返回 `SkillDeclareResult`（`Success` / `Illegal` / `Rejected`）
- `ETryDeclare(skillName, param)`：以 Enemy 视角检查技能合法性
- `Declare(skillName, param, esn, ep)`：真正执行一回合
- `DeepCopy()`：深度复制（重放式复制，见下文）
- `Swap()`：交换双方视角

**DeepCopy 的实现**不是典型的字段级深拷贝，而是：
1. 创建一个全新的 `GameInstance`
2. 重新遍历 `History.SkillHistory`
3. 逐回合调用 `Declare(...)` 重放历史

这意味着：
- AI 模块的模拟局面复制依赖历史声明能确定性地重建状态
- 如果后续加入强随机行为，这个设计需要额外处理随机源

### 6.2 `GameHistory`

`GameHistory` 只保留核心的 `SkillHistory`：
- `List<(ISkillContext, ISkillContext)> SkillHistory`：每一回合双方的原始声明上下文

`Swap()` 方法会把双方历史对调，服务于从另一方视角重组历史的场景。

### 6.3 `Community`

`Community` 表示一方阵营：
- `Body Focus`：当前主要角色
- `List<Body> ActorList`：阵营下的所有角色（初始只包含 `Focus`）

当前引擎虽然以"一方一个主角色"为主，但已经预留了"阵营下多个 `Body`"的结构基础。

### 6.4 `Body` 与 `BodyView`

`Body` 继承自 `ClapBody`，是最核心的领域聚合对象。通过组件模式聚合了七个组件，按类型键访问：

```
Body (extends ClapBody)
├── Community       → Get<Community>()    所属阵营
├── Skill           → Get<Skill>()        技能包管理
├── Health          → Get<Health>()       生命值
├── Defense         → Get<Defense>()      防御列表
├── Resource        → Get<Resource>()     资源系统
├── Effect          → Get<Effect>()       效果系统
└── TurnContext     → Get<TurnContext>()  回合决议缓存
```

`Body.GetView()` 返回 `BodyView` 记录：
- `ProfessionNames`：已激活职业包名称（不含 Common）
- `HP` / `MHP`：当前生命和最大生命
- `DefenseView`：防御列表，每项 `(name, power)`，名称取 `DefenseBase` 子类名
- `ResourcesView`：资源列表，每项 `(name, quantity)`
- `FutureAttackView`：延迟攻击列表，每项 `(name, delayRounds, power)`
- `FutureDefenseView`：延迟防御列表，每项 `(name, delayRounds, power)`

`BodyView` 是前端快照的数据来源。各领域组件自行提供 `GetView()` 方法，视图逻辑与领域逻辑在同一文件中。

### 6.5 `Health`

管理 `HP` 和 `MHP`。方法：
- `GainHP(addition)`：增加 HP，不超过 MHP
- `LoseHP(loss)`：减少 HP
- `GainMHP(addition)`：增加最大生命
- `LoseMHP(loss)`：减少最大生命（当 loss >= MHP 时返回 false 表示失败）
- `Init(hp, mhp)`：重设生命值

### 6.6 `Resource`

资源系统通过 `ResourceTemplate` 建立普通资源与 Gold 资源的配对关系。

构造时反射 `ResourceType._enumDict` 获取所有已注册资源名。以 `Gold_` 前缀命名的资源自动与对应的普通资源配对——它们共享同一个 `ResourceTemplate`。

配对逻辑：
- `Iron` 与 `Gold_Iron` 共享模板
- 无配对资源的类型（如 `Space`、`Time`、`Magic`）使用自己作为 CommonType 和 GoldType
- 消耗资源时默认先消耗 Gold 再消耗 Common
- `ifCommonOnly = true` 时只消耗 Common

关键方法：
- `Check(type, need, ifCommonOnly)`：检查资源是否足够
- `Use(type, need, ifCommonOnly)`：消耗资源
- `Gain(type, need)`：获得资源
- `Query(type)`：查询某种资源的 Common 部分
- `QueryAll(type)`：查询 Common + Gold 总和
- `QuerySpecific(type)`：查询 Gold + Common 的总和（遍历所有资源键，排除 Iron/Gold_Iron/Space/Time/Magic 时的通用计算）

`GetView()` 遍历所有注册键，返回 `(name, quantity)` 列表。

### 6.7 `Defense`

`Defense` 维护 `List<DefenseBase>`，实现了 `IUpdatePerRound`：
- `Add(addition)`：尝试合并（`CanMerge && 同Type`），合并失败则追加并排序
- 排序依据：`DefenseBase.Type`（`DefenseType.BEValue`）的优先级
- `Update()`：每回合清理 `IsDead == true` 的防御
- `GetView()`：返回 `(DefenseBase 类名, Power)` 列表

### 6.8 `Effect`

`Effect` 持有 `List<EffectEntity>`，实现了 `IUpdatePerRound`：
- `Add(entity)` / `AddRange(entities)`：添加效果实体
- `Execute(type, body)`：按 `EffectType` 过滤，处理 `DelayTimes` 和 `RemainingTimes`，执行效果
- `Update()`：清理 `RemainingTimes <= 0` 的效果

### 6.9 `TurnContext`

`TurnContext` 继承自 `ClapTurnContext<IResolution, Community>`，注册了四种决议类型：

- `AttackResolution`：攻击意图
- `DefenseResolution`：防御意图
- `ResourceResolution`：资源变化意图
- `EffectResolution`：效果施加意图

每种决议都实现了 `IResolution` 接口：
```csharp
public interface IResolution : IClapResolution<Community>
{
    int DelayRounds { get; set; }
    float Power { get; set; }
}
```

`ExecuteImpl` 实现了统一的延迟处理：只执行 `DelayRounds == 0` 的决议，其余递减 `DelayRounds`。

视图方法：
- `GetFutureAttackView()`：返回所有延迟攻击的 `(攻击类型名, 剩余延迟回合, 力度)`
- `GetFutureDefenseView()`：返回所有延迟防御的 `(防御类型名, 剩余延迟回合, 力度)`

## 7. 技能系统

### 7.1 `Skill` 聚合器

`Skill` 类是角色可用技能的统一入口，内部维护一个 `Package` 列表。

每个 `Package` 包含：
- `Name`：技能包类名
- `SkillPackage`：`ISkillPackage` 实例
- `IsActive`：是否激活

默认构造时会自动装入一个 `Common` 实例作为第一个包。这意味着所有角色开局至少拥有 `Common`。

主要方法：
- `AddPackage(skillPackage)`：追加技能包
- `AddSkill(packageName, skillName)` / `RemoveSkill(packageName, skillName)`：动态增删技能名
- `TryDeclare(skillName, sc)`：按包的激活顺序查找技能，返回三种结果
- `Declare(skillName, sc)`：调用对应技能包的 `SkillSourceFileGenerator[skillName]` 生成 DSL
- `GetPassiveSkill(sc)`：收集所有包的被动技能 DSL
- `GetAvailableSkillNames()`：收集所有激活包的可用技能名
- `GetView()`：返回除 Common 以外的职业包名称列表
- `HaveProfession`：`_packages.Count > 1`

### 7.2 技能声明语义

`TryDeclare` 的返回值三种：

- **`Success`**：技能名存在且校验函数返回 `true`
- **`Rejected`**：技能名存在但校验函数返回 `false`（资源不足或前置条件不满足）
- **`Illegal`**：在任何激活包中都不存在该技能名

这个区分对调试很有用——能判断是"技能未注册"还是"资源不足"。

### 7.3 `SkillPackageBase`

`SkillPackageBase` 继承自 `ClapInfra.ClapProfession.ClapSkillPackage<ISkillContext, IDSLSourceFile>`，是 Blacksmith 所有技能包的基类。

反射配对逻辑由 `ClapSkillPackage` 基类完成：自动扫描 `private` 实例方法，将 `XxxCheck` 和 `Xxx` 配对为技能。配对要求 `Xxx` 方法返回 `IDSLSourceFile`（而非 `DSL.SourceFile`），技能名统一转为小写。

`SkillPackageBase` 提供了默认的 `PassiveSkill` 实现（返回空的 `SourceFile`，它实现了 `IDSLSourceFile` 接口）。

### 7.4 `MainProfession` 与 `ProfessionModifier`

- `MainProfession`：标记 `PackageType = PackageType.Main`。在主职业包构造函数中会调用 `AddModOnInit()` 合并修改器。
- `ProfessionModifier`：应在子类上标注 `[IsProfessionModifier("目标职业名")]`。定义 `PackageType = PackageType.Modifier`。

典型扩展模式：
1. 写一个新的 `MainProfession`
2. 写一个 `[IsProfessionModifier(nameof(Common))]` 的修改器
3. 在修改器中提供转职技能入口，内部调用 `AddPackage(new YourProfession())` 和 `Common.ExcludeAllProfessions(source)` 清理其他转职入口

### 7.5 `ProfessionRegistry`

维护两类信息：
- `Professions`：所有主职业名的列表
- `_modifiers`：`Dictionary<string, List<SkillPackageBase>>`，每个目标职业对应的修改器列表

`AddModOnInit(package)` 在包构造时被调用：查找所有目标为该包名的修改器，通过反射创建新实例，将其技能（技能名、Checker、Generator）复制到目标包。

### 7.6 内置职业设计概览

**Common**：基础技能包，包含：
- 资源获取：`iron`、`space`、`time`
- 物理攻击：`stick`(1)、`drill`(3)、`slash`(5)
- 撕裂：`tear`（消耗 1 Space，8 ATK）
- 防御：`shield`（可参数化）、`thornshield`（刺盾）
- 恢复：`recovery`（可参数化）
- 反射：`reflect`（动态规则，消耗 2 Space）
- 六条转职入口：`warlock`、`cannon`、`driver`、`bloodsigil`、`lancer`

**Warlock**（术士）：魔法职业
- `magic`：1 Iron → 1 Magic
- `magicattack(n)`：消耗 n Magic，三回合延迟物理攻击（每回合 2n ATK）
- `mute`：禁言，阻止对手本回合 Space/Time 获取
- `sacrifice`：消耗 1 HP + 1 MHP，获得 7 RDEF + 1.5 Iron
- `alchemy`：消耗 2.5 Iron，解锁 `midastouch` 并移除此技能
- `midastouch`：消耗 1 Iron（仅 Common），产出 5 Gold_Iron

**Cannon**（钢炮）：高伤害物理职业
- `strike`：1 Iron → 4 ATK
- `doublestrike`：2 Iron → 8 ATK
- `triplestrike`：3 Iron → 11 ATK
- `apshell`（穿甲弹）：1 Iron → 2 ATK（AP 因子 3）+ 打断
- `cannonbarrel`（炮管）：0 → 2 DEF + 1 ATK

**Driver**（驱动器）：时空资源职业
- 被动：每回合自动获得 `1 + min(5, Time数量 * 2)` RDEF
- `spaceattack(n)`：消耗 n Space → 12n ATK
- `space2time`：1 Space → 1 Time + 3 RDEF
- `time2space`：1 Time → 1 Space + 3 RDEF
- `spacebarrier(n)`：消耗 n Iron → (5.5n - 0.5n²) RDEF（n ≤ 5）

**BloodSigil**（鲜血印记）：生命值驱动职业
- 转职时：+3 MHP，+3 HP，移除 `stick`/`drill`/`slash`/`tear`
- `bloodblade`（血刃）：消耗 4 HP → 6 ATK + 66% 吸血
- `bloodlust`（嗜血）：消耗 2 HP → 下次攻击伤害 ×1.5
- `bloodrecovery`（恢复）：无需 HP → 回复 1 HP
- `bloodshield`（血之盾）：消耗 1 HP → DEF = HP×40%
- `bloodrage`（狂怒）：HP 在 2~5 之间可用，消耗 1 HP → 5 ATK + 150% 吸血

**Lancer**（战矛）：纹章系统职业
- 四种基础攻击命中获得对应纹章，下次攻击消耗并附加效果：
  - `skystrike`（天击）：3 ATK + 打断 → 火纹（下次攻击 +2 伤害）
  - `dragontooth`（龙牙）：3 ATK + 3 DEF → 冰纹（下次攻击 +2 AMR）
  - `tyrantdestruction`（霸碎）：3 ATK + AP=2 → 光纹（下次攻击 +2 HP）
  - `triplestab`（连突）：2+2+1 ATK 三段 → 暗纹（下次攻击加 1 RATK × 2 回合）
- `risingdragon`（伏龙翔天）：4 Iron → 9 + 蓄力计数×4 MATK
- `charge`（蓄力）：4 Iron → 蓄力计数+1，下次蓄力/伏龙翔天免费。若被攻击则自动触发伏龙翔天（通过 `Charge` 动态规则实现）

## 8. 技能 DSL

### 8.1 为什么需要 DSL

技能不是直接修改战场状态，而是先翻译为中间表示（`IDSLSourceFile`，具体实现为 `DSL.SourceFile`）。这层中间表示带来的收益：
- 技能定义与判定流程解耦
- 技能可以先"写意图"，再由统一判定链消费
- 同一技能可以组合攻击、防御、资源、效果、自由动作
- 动态规则可以在编译阶段挂入

`SourceFile` 实现了 `IDSLSourceFile` 接口，该接口来自 ClapInfra 的 `IClapDSLSourceFile<...>` 泛型契约，只暴露 `Compile(Judger?)` 一个方法。

### 8.2 `SourceFile` 的内部结构

`SourceFile` 是一个脚本构造器，内部维护三类数据：

- `_sentences`：主语句列表（`List<Sentence>`）。每条 `Sentence` 包含：
  - `Structure`：`Action<Community>` 执行体
  - `SentenceType`：`Attack / Defense / Resource / Effect / Recovery / Free`
  - `StructureType`：`Main`（主语句）或 `Rhetoric`（修辞语句）
  - `BindSentence`：修辞依附的主语句引用（仅修辞语句有值）

- `_rhetoricCache`：修辞缓存（`Stack<Sentence>`）。用栈而不是列表是为了在编译时按"后进先插入"的顺序展开。

- `_mutationsOnCompile`：编译时要注册的动态规则字典。

### 8.3 Pen 模式

技能 DSL 使用函数式 `Pen` 来实现流畅链式调用：

```csharp
using Pen = Func<DSLforSkillLogic.SourceFile, DSLforSkillLogic.SourceFile>;

// 使用示例
Pen pen = sf => sf
    .UseResource(1, ResourceType.Instance.Iron())
    .WriteAttack(3, AttackType.Instance.Physical())
    .WithBloodSuck(0.66f);

return DSL.Create(sc.Self, pen);
```

`DSL.Create(community, pen)` 创建一个以 `community` 为 owner 的 `SourceFile`，然后应用 `pen`。

### 8.4 DSL 主要语句

#### `WriteAttack(power, attackType, APFactor, delayRounds)`

往 `TurnContext` 写入 `AttackResolution`。返回 `AttackFile`（继承 `SourceFile`）以支持链式调用修辞。

`AttackResolution.Execute` 的执行流程：
1. 若为真实伤害，跳过防御直接扣 HP
2. 否则按优先级顺序遍历防守方全部防御
3. 遇到穿甲列表中的防御类型（`ThornReduction`、`CommonReduction`、`RockArmor`、`ReadlArmor`、`CommonArmor`）时伤害乘以 `APFactor`
4. 调用每个防御的 `Work(source, owner, attack, type)` 计算穿透后的剩余伤害和已被吸收的伤害
5. 穿透完毕后乘以 APFactor 还原（`Ceiling`）
6. 最终剩余伤害从目标 `Health.HP` 扣除
7. 阶段钩子：首次命中护甲时触发 `OnHitArmorFirstTime`，结算完毕触发 `OnEnd`

#### `WriteDefense(power, defense, delayRounds)`

`DefenseResolution.Execute` 将 `defense.Power` 设为 `power`，然后 `target.Get<Defense>().Add(defense)`。

#### `WriteResource(power, resourceType, delayRounds)`

`ResourceResolution.Execute` 调用 `target.Get<Resource>().Gain(type, power)`。

#### `WriteEffect(type, targetType, power, duration, effectAction)`

`EffectResolution.Execute` 创建一个 `EffectEntity` 并添加到目标角色上。配合 `effectAction` 参数定义效果触发时的具体逻辑。

#### `WriteRecovery(power)`

直接调用 `source.Focus.Get<Health>().GainHP(power)`。

#### `WriteFree(action)`

写入任意自定义动作。这是最灵活的 DSL 语句，也是最容易绕开统一规则链的地方。

#### `UseResource(need, type, ifCommonOnly)`

语法糖，通过 `WriteFree` 调用 `Resource.Use(type, need, ifCommonOnly)`。

### 8.5 攻击修辞（仅 `AttackFile`）

- `WithBloodSuck(percent)`：攻击结算完毕时，按 `power * percent` 向上取整回复攻击方 HP。挂载在 `AttackStage.OnEnd`。
- `WithInterupt()`：攻击首次命中护甲时，移除目标 TurnContext 中所有 Iron/Gold_Iron/Magic 类型的 `ResourceResolution`。挂载在 `AttackStage.OnHitArmorFirstTime`。
- `WithFree(stage, action)`：通用修辞，在指定攻击阶段执行任意动作。

### 8.6 编译：`Compile(judger)`

1. 展开修辞缓存：将修辞语句插入到其 `BindSentence` 之后
2. 如果有 `judger`：遍历 `_mutationsOnCompile`，依次调用 `RegistJudgeRuleDynamic` 和 `AddJudgeRule`
3. 合并所有语句的 `Structure` 为一个复合 `Action<Community>`
4. 返回 `Intent { Execute = 合并后的 Action }`

## 9. 从声明到结算：一个回合的完整链路

### 9.1 `GameInstance.Declare(skillName, param, esn, ep)`

1. 为双方创建 `DefaultSkillContext`
2. 将本回合声明追加到 `History.SkillHistory`
3. 调用 `Player.Focus.Get<Skill>().Declare(skillName, playerContext)` 生成玩家主动技能 `SourceFile`
4. 调用 `GetPassiveSkill(playerContext)` 获取被动技能列表，插入到主动技能前面
5. 对敌方做同样处理
6. 调用 `Judger.Judge(psfs, esfs)`

每方不是只有一个 `SourceFile`，而是"多个被动 + 一个主动"的 `SourceFile` 列表。

### 9.2 `Judger.Judge(playerSfs, enemySfs)`

`Judger` 继承自 `ClapInfra.ClapJudgement.ClapJudger<...>`，其 `Judge()` 方法编排判定主循环。Blacksmith 的 `Judger` 重写了 `Compile()`：

1. `Compile(playerSfs)`：编译玩家技能列表
   - 第一个元素（被动技能汇总）单独 `Compile`
   - 其余元素逐个 `Compile`，被动技能的 Intent 插回结果最前面
2. `Compile(enemySfs)`
3. 触发 `_onJudge`：先 `TranslateIntentsToResolutions`，然后执行规则链

### 9.3 `TranslateIntentsToResolutions`

依次执行所有玩家 Intent → 所有敌方 Intent。每个 `Intent` 继承自 `ClapInfra.ClapJudgement.Core.ClapIntent<Community>`，其 `Execute` Action 通常向施法者的 `TurnContext` 写入各种 Resolution。

### 9.4 `JudgeRuleManager.GetRule()`

`JudgeRuleManager` 继承自 `ClapInfra.ClapJudgement.ClapJudgeRuleManager<Community>`，按 `JudgeStage.BEValue` 优先级升序遍历所有阶段容器，为每阶段调用 `Build()` 生成规则链，累加为一个复合 `Action<Community, Community>`，然后执行。

## 10. 判定引擎与阶段链

### 10.1 `JudgeRuleManager`

维护 `SortedDictionary<JudgeStage.BEValue, StageRuleContainer>`。

每个 `StageRuleContainer` 包含：
- 一个基础规则 `_baseRule`
- 多个 `RuleUnit`：`_overrideRules`、`_modifiersBefore`、`_modifiersAfter`

`Build()` 方法：
1. `Update()`：清理过期规则，推进延迟/剩余回合
2. 选出 `RemainingRounds == 0` 的规则
3. 返回 `Before Modifiers → 核心规则(Override 或 Base) → After Modifiers`

### 10.2 阶段顺序与默认规则

| 阶段 | 优先级 | 默认规则 |
|---|---|---|
| `OnBegin` | 0 | 空操作 |
| `OnEffectTaking_AfterResolutionWritten` | 8 | 双方执行 `AfterResolutionWritten` 类型效果 |
| `OnEffectSwaping` | 16 | 交换双方当回合/指向敌方的 `EffectResolution` |
| `OnAttackCanceling` | 32 | 按优先级顺序对消双方攻击（跳过真实伤害和延迟攻击） |
| `OnAttackSwaping` | 64 | 交换双方当回合 `AttackResolution` |
| `OnApplyingEffect` | 128 | 双方执行 `EffectResolution` |
| `OnEffectTaking_AfterTransport` | 256 | 双方执行 `AfterTransport` 类型效果 |
| `OnApplyingOthers` | 512 | 防御 → 攻击 → 资源（依次 Execute） |
| `OnUpdating` | 1024 | 双方所有 `Body.ActorList` 调用 `Update()` |
| `OnEffectTaking_AfterResult` | 2048 | 双方执行 `AfterResult` 类型效果 |
| `OnEnd` | 4096 | 空操作 |

阶段之间的优先级差值（8、16、32...）为 Mod 插入新阶段预留了空间。

### 10.3 攻击对消（`CancelAttacks`）

双方按攻击类型优先级排序（`Real(0) > Magical(128) > Physical(256)`），依次 1:1 抵消。跳过真实伤害和延迟攻击。对消后攻击力归零的从列表中移除。

### 10.4 效果/攻击交换

交换逻辑是把一方"指向敌方"或"当回合生效"的 Resolution 移到对方列表。这意味着 DSL 中写的攻击总是先写到自己一侧，再由阶段链交换到对方那里——这是规则链对称性的保证。

## 11. 动态规则系统

### 11.1 为什么需要动态规则

普通 DSL 擅长描述"造成伤害、增加防御、消耗资源、增加效果"，但不擅长描述"改写判定流程本身"的技能（反射、转移、延迟保护、自动反击）。动态规则系统为此提供了在已有阶段链上做局部改写的能力。

### 11.2 参与的核心类型

- **`DynamicJudgeRuleName`**：可扩展枚举，标识规则类别（`Reflect`、`Charge`）
- **`Mutation`**：描述一条具体的规则改变（阶段、类型、顺序、持续时间）
- **`MutationPrototype`**：`Mutation` 的模板版本，因为动态规则通常需要知道"是谁触发的"
- **`DynamicJudgeRulePool`**：存储和查询模板的池
- **`JudgeRuleManager`**：消费专门化后的规则

### 11.3 完整链路

1. 技能 DSL 调用 `LinkJudgeRuleDynamic(ruleKey, mutations)`
2. `SourceFile.Compile(judger)` 时调用 `judger.JudgeRuleManager.RegistJudgeRuleDynamic(ruleKey, mutations)`
3. `DynamicJudgeRulePool.RegistDynamic` 将 `Mutation` 转换为 `MutationPrototype`
4. `AddJudgeRule(source, ruleKey)` 调用 `DynamicJudgeRulePool.Query(source, ruleKey)` 按施法者专门化
5. 专门化后的 `Mutation` 按 `Stage` 插入对应 `StageRuleContainer`

### 11.4 按施法者专门化

默认规则通常是对称的（双方处理相同），但像 `reflect` 这种技能不是纯对称规则——它必须知道是谁发起的。`MutationPrototype` 保存三参数版本的规则（`Action<Community source, Community player, Community enemy>`），`Specialize(source)` 时固化 `source`，产出标准的二参数规则。

### 11.5 `RuleUnit` 的生存期

`RuleUnit` 有两个时间维度：
- `DelayRounds`：还需等待几回合才开始减少 `RemainingRounds`
- `RemainingRounds`：规则还能执行几次

每轮 `TimePass()` 时先减少 `DelayRounds`，归零后才减少 `RemainingRounds`。`Build()` 只选择 `RemainingRounds == 0` 的规则参与执行。

## 12. 防御系统

### 12.1 `DefenseBase` 抽象类

所有防御类型都继承自 `DefenseBase`：
- `Type`：`DefenseType.BEValue`，决定防御的排序优先级和分类
- `Power`：防御力度
- `CanMerge`：同类型防御是否可合并
- `IsDead`：是否已失效（下回合清理）
- `Work(source, owner, attack, type)`：处理攻击，返回 `(剩余攻击力, 已吸收伤害)`
- `Update()`：每回合状态更新

### 12.2 内置防御类型

| 防御类型 | Type 枚举 | 优先级 | 特性 |
|---|---|---|---|
| `RealReduction` | `RealReduction` | 0 | 真实伤减，单回合失效 |
| `ThornReduction` | `ThornReduction` | 8 | 刺盾，单回合失效。吸收物理攻击时，反弹 50%（向上取整）为魔法伤害（延迟 1 回合） |
| `CommonReduction` | `CommonReduction` | 16 | 普通伤减，单回合失效 |
| `RockArmor` | `RockArmor` | 32 | 岩甲（枚举已定义，暂无实现） |
| `ReadlArmor` | `ReadlArmor` | 64 | 真实装甲（枚举已定义，暂无实现） |
| `CommonArmor` | `CommonArmor` | 128 | 普通装甲，耐久型，Power 随吸收伤害递减 |

优先级值越小越先被攻击遍历到。真实伤减最先处理，普通装甲最后处理。

### 12.3 `AttackResolution` 中的防御处理

攻击穿透防御时有一个 AP 因子机制：在遍历穿甲列表中的防御类型时，攻击力临时乘以 `APFactor`（默认 1，穿甲弹设为 3）；穿透该防御后除以 `APFactor` 还原。这实现了"穿甲弹对非真实防御具有三倍穿透力"的效果。

## 13. AI 模块

### 13.1 `IAIStrategy` 接口

```csharp
public interface IAIStrategy
{
    string Name { get; }
    void Init(GameInstance gameInstance);
    (string skillName, int param) ChooseSkill(Community self, Community opponent);
}
```

AI 的接入位置在会话层之上、规则层之下：它不负责启动会话，也不直接改写判定引擎，只是替敌方提供本回合声明。

### 13.2 `BloodSigilStrategy`

基于规则的启发式 AI。特点：
- 前 7 回合固定积攒 `iron`
- 第 8 回合转职 `bloodsigil`
- 之后根据血量分档决策：HP≤1 → recovery，HP≤5 → rage，否则按对手职业分支
- 对 Cannon 和 Driver 有不同的决策分支，评估"当前是否可能被击杀"

### 13.3 `GeneralStrategy`

基于 MCTS（蒙特卡洛树搜索）的通用 AI。核心特征：
- 通过 `DeepCopy()` 复制游戏状态进行模拟
- 多线程并行搜索（最多 8 线程）
- 使用 UCT 公式进行节点选择
- 评分函数综合考虑：资源（Iron/Space/Time/Magic 分阶段加权）、职业状态、HP 差值、回合节奏
- 支持参数化（`GeneralStrategyParams`），可变异和交叉，为参数进化做准备
- 最终用软最大化（temperature）从 top-K 动作中采样

## 14. 可扩展枚举体系

### 14.1 `BlacksmithEnum<T>`

继承自 `ClapEnum<T, IsBlacksmithEnumMember>`。`[IsBlacksmithEnumMember(priority)]` 特性的 `priority` 参数决定排序顺序。

### 14.2 当前内置可扩展枚举一览

| 枚举类型 | 成员（优先级） | 说明 |
|---|---|---|
| `ResourceType` | Iron(0), Gold_Iron(1), Space(2), Time(3), Magic(4) | Gold_Iron 与 Iron 配对 |
| `AttackType` | Real(0), Magical(128), Physical(256) | 优先级决定对消顺序 |
| `DefenseType` | RealReduction(0), ThornReduction(8), CommonReduction(16), RockArmor(32), ReadlArmor(64), CommonArmor(128) | 优先级决定攻击遍历顺序 |
| `EffectType` | AfterResolutionWritten(0), AfterTransport(8), AfterResult(16) | 对应不同判定阶段的触发时机 |
| `EffectTargetType` | Self(0), Enemy(8) | 效果目标 |
| `DynamicJudgeRuleName` | Reflect(0), Charge(1) | 动态规则键名 |
| `JudgeStage` | OnBegin(0), AfterResolutionWritten(8), OnEffectSwaping(16), OnAttackCanceling(32), OnAttackSwaping(64), OnApplyingEffect(128), AfterTransport(256), OnApplyingOthers(512), OnUpdating(1024), AfterResult(2048), OnEnd(4096) | 判定阶段 |

### 14.3 枚举扩展方法的约定

带 `[IsBlacksmithEnumModifier]` 的静态类中，可以写：
- 带 `[IsBlacksmithEnumMember(priority)]`
- 只有一个参数（`this EnumType`）
- 返回 `BEValue`

的方法来新增或覆盖枚举成员。方法名成为成员名，同名方法覆盖优先级。

## 15. 前端快照模型

### 15.1 分布式 `GetView()`

快照构造由各领域对象自行提供 `GetView()` 方法：
- `Body.GetView()` → `BodyView`
- `Defense.GetView()` → `List<(string name, int power)>`
- `Resource.GetView()` → `List<(string name, float quantity)>`
- `TurnContext.GetFutureAttackView()` → `List<(string name, int delayRounds, int power)>`
- `TurnContext.GetFutureDefenseView()` → `List<(string name, int delayRounds, int power)>`
- `Skill.GetView()` → `List<string>`（职业名列表，不含 Common）

`WebGameSession.BuildSnapshot()` 直接组装这些视图数据为匿名对象返回。

### 15.2 快照 JSON 结构

```json
{
  "started": true,
  "modeName": "Blood Sigil",
  "manualMode": false,
  "result": "next",
  "player": {
    "professions": ["BloodSigil"],
    "hp": 10, "maxHP": 13,
    "defenses": [{"name": "CommonReduction", "power": 3}],
    "resources": [{"name": "Iron", "quantity": 5}],
    "futureAttacks": [{"name": "Magical", "delayRounds": 1, "power": 2}],
    "futureDefenses": [],
    "availableSkills": ["iron", "bloodblade", ...]
  },
  "enemy": { /* 同 player 结构 */ },
  "turns": [{
    "index": 1, "result": "Continue",
    "playerSkill": "iron", "playerParam": 0,
    "enemySkill": "iron", "enemyParam": 0
  }]
}
```

### 15.3 可声明技能列表

`Skill.GetAvailableSkillNames()` 直接返回所有激活包中的所有可用技能名，不做参数试探过滤。技能是否真正可用（资源是否足够）由 `TryDeclare` 在声明时校验。

### 15.4 快照扩展性

由于 `GetView()` 遍历内部完整数据结构，新增资源/防御类型会自动反映在快照中。但前端需要额外配合的部分包括：技能说明文本、特殊 UI 渲染逻辑。

## 16. 典型扩展路径

### 16.1 新增一个普通职业

1. 新建继承 `MainProfession` 的类，用 `XxxCheck` / `Xxx` 模式定义技能
2. 新建继承 `ProfessionModifier` 的类，加 `[IsProfessionModifier(nameof(Common))]`
3. 在修改器中写一个转职技能，`AddPackage(new YourProfession())` + 清理入口
4. 编译 DLL 放到宿主运行目录

### 16.2 新增一种资源

1. 写带 `[IsBlacksmithEnumModifier]` 的静态类
2. 为 `ResourceType` 增加新成员
3. 若有 Gold 配对，命名遵循 `Gold_Xxx`
4. 在技能 DSL 中用 `WriteResource` / `UseResource` 使用
5. 前端展示由 `Resource.GetView()` 自动处理

### 16.3 新增一种防御类型

1. 在 `DefenseType` 扩展枚举中注册新成员（注意优先级）
2. 继承 `DefenseBase` 实现防御行为（`Work`、`Update` 等）
3. 在技能 DSL 中通过 `WriteDefense(power, new YourDefense())` 使用

### 16.4 新增动态规则

1. 确保不能只用普通 DSL 实现
2. 扩展 `DynamicJudgeRuleName` 枚举（如果需要新的键名）
3. 准备 `Mutation` 列表
4. 在技能 DSL 中调用 `LinkJudgeRuleDynamic`

## 17. 维护与排障指南

### 前端显示不对
先看 `wwwroot/js/*`、`Body.GetView()`、`WebGameSession.BuildSnapshot()`。检查字段名是否匹配。

### 技能存在但按钮不可用
区分 `Illegal`（技能未注册）和 `Rejected`（`XxxCheck` 校验不通过）。先查 `GetAvailableSkillNames()` 确认技能已注册，再检查对应校验函数。

### 技能未出现在可用列表中
最常见原因：技能生成方法的返回类型写了 `DSL.SourceFile` 而非 `IDSLSourceFile`。`ClapSkillPackage` 反射配对要求方法签名精确匹配 `Func<ISkillContext, IDSLSourceFile>`，返回具体类会导致配对失败、技能静默缺失。

### 技能结算结果不对
按顺序排查：
1. `SkillPackageBase` 是否找到正确的生成方法（检查反射配对——方法是否返回 `IDSLSourceFile`）
2. `SourceFile` 是否包含预期的语句
3. `Compile` 时是否注册了预期的动态规则
4. `TurnContext` 中是否有预期的 `Resolution`
5. `JudgeRuleManager` 各阶段顺序和规则是否正确

### Mod 没生效
1. DLL 是否在宿主运行目录？
2. 类型名是否与已注册类型冲突？
3. 特性是否正确标注？
4. 枚举扩展是否在 `CloseFactory` 前完成？
5. 修改器的目标职业名是否匹配？

### AI 行为异常
检查 `ChooseSkill` 输出是否通过 `TryDeclare` 验证，检查 `DeepCopy` 重放状态是否准确。

## 18. 一句话总结

Blacksmith 当前最核心的架构思想是：把"技能描述""回合上下文""阶段化判定""动态规则改写""本地 Web 会话"拆成独立层，再由 `BlacksmithClient` 在启动时装配成一个可运行、可调试、可扩展的本地对战宿主。

核心数据流：
```
SkillPackage → IDSLSourceFile → Compile(Judger) → Intent → TurnContext(Resolutions) → JudgeRuleManager(Stage Chain) → Body.GetView()
```

其中 `ClapJudger`（ClapInfra）提供判定主循环骨架，`Judger`（BlacksmithCore）处理被动技能顺序；`IClapDSLSourceFile`（ClapInfra）定义编译契约，`SourceFile`（BlacksmithCore）实现具体 DSL。

这条链就是从"玩家声明一个技能"到"前端看到回合结果"的主干路径。
