# 项目架构

本文档面向以下三类读者：

- 想继续维护 `Blacksmith` 的开发者
- 想定位规则、回合结算或前端展示问题的调试者
- 想基于当前引擎编写大型 Mod、职业包或 AI 的扩展开发者

目标不是只讲项目有哪些目录，而是尽量把下面几件事讲清楚：

- 程序是如何启动并完成装配的
- 一局游戏在内存中由哪些核心对象组成
- 一个技能从“声明”到“真正生效”经历了哪些阶段
- Web 宿主、前端快照、AI、Mod 分别插在哪一层
- 当你需要扩展或排障时，应该先看哪里

---

## 1. 总体设计思路

当前仓库可以概括为“单宿主 + 核心库 + 示例扩展”三层结构：

- `Blacksmith/BlacksmithClient`
  运行入口。负责加载插件、初始化 AI、启动本地 Web 宿主、暴露最小 API、托管静态前端。
- `Blacksmith/BlacksmithCore`
  真正的游戏引擎。包括领域模型、技能系统、DSL、判定引擎、动态规则、AI 接口、插件基础设施、Web 会话封装。
- `Blacksmith/ModExamples`
  示例扩展目录。用于演示“扩展资源/防御枚举 + 新职业 + Common 修改器”的写法。

和传统“前后端分离”的项目不同，这个仓库当前不是一个远程服务端加独立前端工程，而是：

1. 由 `BlacksmithClient` 启动 ASP.NET Core 本地站点
2. 用 `wwwroot` 托管一个轻量静态页面
3. 用几个 `/api/*` 端点操作同一进程里的 `GameInstance`

也就是说，前端、会话层、规则引擎默认跑在同一个进程里。

这样做的直接结果是：

- 启动成本低，本地调试简单
- 规则对象不需要额外序列化成复杂服务协议
- 但 Web 层与引擎层的边界要靠“快照对象”和“会话门面”人为保持清晰

## 2. 解决方案与目录结构

当前解决方案文件为 `Blacksmith/Blacksmith.sln`，其中包含三个项目：

- `BlacksmithCore`
- `BlacksmithClient`
- `ModExamples`

### 2.1 `BlacksmithClient`

项目路径：`Blacksmith/BlacksmithClient`

目标框架：`net8.0`

职责：

- 作为整个应用的进程入口
- 扫描运行目录中的 DLL 并触发插件注册
- 构造内置 AI 策略列表
- 启动本地 Web 宿主
- 暴露 `/api/strategies`、`/api/start`、`/api/declare`、`/api/status`
- 托管 `wwwroot` 下的静态资源

关键文件：

- `Blacksmith/BlacksmithClient/Program.cs`
- `Blacksmith/BlacksmithClient/Frontend/LocalHost.cs`
- `Blacksmith/BlacksmithClient/Frontend/WebGameSession.cs`
- `Blacksmith/BlacksmithClient/wwwroot/index.html`
- `Blacksmith/BlacksmithClient/wwwroot/css/style.css`
- `Blacksmith/BlacksmithClient/wwwroot/js/api.js`
- `Blacksmith/BlacksmithClient/wwwroot/js/main.js`
- `Blacksmith/BlacksmithClient/wwwroot/js/state.js`
- `Blacksmith/BlacksmithClient/wwwroot/js/types.js`
- `Blacksmith/BlacksmithClient/wwwroot/js/ui.js`

### 2.2 `BlacksmithCore`

项目路径：`Blacksmith/BlacksmithCore`

目标框架：`net8.0`

职责：

- 定义游戏的核心领域对象
- 提供技能声明与技能实现机制
- 用 DSL 把技能描述转成可执行意图
- 组织回合判定阶段和默认规则链
- 支持通过动态规则改写某些判定阶段
- 提供插件加载、可扩展枚举、职业注册等扩展基础设施
- 提供 AI 接口与内置 AI
- 通过 `BodyView` 及各类 `GetView()` 方法向 Web 层输出只读快照数据

核心子目录：

- `AI`
- `Backend/JudgementLogic`
- `Backend/SkillPackages`
- `FrontEndBackendInterface`
- `Infra`

### 2.3 `ModExamples`

项目路径：`Blacksmith/ModExamples`

职责：

- 演示如何新增 `ResourceType` / `DefenseType` 成员
- 演示如何实现新的 `MainProfession`
- 演示如何通过 `ProfessionModifier` 把新职业挂到 `Common`

定位建议：

- 它更适合作为“扩展模式参考”
- 不建议把它当成当前 API 的唯一权威来源

## 3. 运行时分层

如果从职责而不是目录来分，可以把系统理解成下面六层：

### 3.1 宿主层

代表代码：

- `Program`
- `LocalHost`

职责：

- 进程启动
- 插件装配
- AI 装配
- HTTP 路由
- 静态资源服务

### 3.2 会话层

代表代码：

- `WebGameSession`
- `BackendStarter`

职责：

- 创建或重置游戏局
- 维护当前模式和 AI
- 校验 API 输入
- 根据模式决定敌方行动来源
- 调用领域对象的 `GetView()` 方法，组装前端快照

### 3.3 应用编排层

代表代码：

- `GameInstance`
- `GameHistory`

职责：

- 持有一局游戏的核心对象
- 接收双方本回合声明
- 组织被动技能与主动技能的合并
- 调用判定引擎
- 记录回合历史

### 3.4 领域模型层

代表代码：

- `ActorSet`
- `Body` / `BodyView`
- `Health`
- `Resource`
- `Defense`
- `Effect`
- `TurnContext`
- 各类 `Resolution`

职责：

- 表达角色状态
- 暂存本回合要发生的行为
- 提供资源、防御、效果等基础规则载体
- 通过 `GetView()` 方法向外暴露只读快照数据

### 3.5 规则执行层

代表代码：

- `SkillPackageBase`
- `DSLforSkillLogic`
- `Intent`
- `Judger`
- `JudgeRuleManager`
- `DynamicJudgeRulePool`

职责：

- 从技能名和上下文生成“技能脚本”
- 把脚本编译成意图
- 把意图写入 `TurnContext`
- 按阶段执行默认规则与动态规则

### 3.6 扩展层

代表代码：

- `PluginLoader`
- `BlacksmithEnum<T>`
- `BlacksmithEnumRegistry`
- `ProfessionRegistry`
- `IsBlacksmithEnumModifier`
- `IsProfessionModifier`

职责：

- 扫描运行目录的程序集
- 注册扩展枚举成员
- 注册新职业与职业修改器
- 让运行时在启动阶段完成“扩展装配”

## 4. 启动链路

启动入口位于 `Blacksmith/BlacksmithClient/Program.cs`。

整体顺序如下：

1. `PluginLoader.Initialize(AppContext.BaseDirectory)`
2. `LoadBlacksmithEnumModifierPlugins()`
3. `LoadProfessionPlugins()`
4. 初始化 AI 策略
5. `LocalHost.Start(strategies)`

### 4.1 插件扫描

`PluginLoader.Initialize` 会扫描运行目录下的所有 `.dll`，并尝试 `Assembly.LoadFrom`。

这一步只做两件事：

- 读取程序集
- 放入内部缓存 `_cache`

它不会在这一步立刻完成所有注册，真正的“按类型实例化”和“按特性识别静态扩展类”发生在后面。

### 4.2 扩展枚举注册

`Program.LoadBlacksmithEnumModifierPlugins()` 分两段执行：

1. `PluginLoader.LoadPluginsByType<BlacksmithEnum>()`
   扫描所有 `BlacksmithEnum` 的具体子类并实例化，然后通过 `BlacksmithEnumRegistry.RegistBlacksmithEnum` 注册。
2. `PluginLoader.LoadBlacksmithEnumModifierPlugins()`
   扫描带 `[IsBlacksmithEnumModifier]` 特性的静态类，再读取其中带 `[IsBlacksmithEnumMember]` 的扩展方法，把这些方法对应的新成员注册到目标枚举中。

最后调用 `BlacksmithEnum.CloseFactory()`，关闭扩展枚举的工厂能力。

这意味着：

- 可扩展枚举的“定义期”只在启动阶段开放
- 一旦宿主完成启动，后续运行中就不再允许动态增补这些枚举成员

### 4.3 职业与修改器注册

`Program.LoadProfessionPlugins()` 也分两段：

1. 先扫描 `SkillPackageBase` 子类
2. 第一轮注册所有 `PackageType.Main` 的职业名
3. 第二轮注册所有 `PackageType.Modifier` 的职业修改器

这样安排的原因是：

- `Common` 一类职业包在初始化时可能需要知道有哪些可转职职业名
- 修改器需要明确自己挂载到哪个目标职业上

### 4.4 AI 初始化

当前宿主默认创建两种 AI：

- `BloodSigilStrategy`
- `GeneralStrategy`

其中 `GeneralStrategy` 会尝试从 `data.json` 读取参数；若读取失败，则回退为默认参数。

### 4.5 Web 宿主启动

最终调用 `LocalHost.Start(strategies)`：

- 创建 ASP.NET Core 应用
- 绑定 `http://localhost:5000`
- 启用默认首页和静态文件
- 构造 `WebGameSession`
- 映射 API
- 自动打开浏览器
- 阻塞运行

## 5. Web 宿主与 API 层

### 5.1 `LocalHost` 的职责边界

`BlacksmithClient/Frontend/LocalHost.cs` 是一个非常薄的宿主层。

它做的事有：

- 配置 `WebRootPath` 与 `ContentRootPath`
- 固定监听 `http://localhost:5000`
- 挂载静态文件中间件
- 创建一个 `WebGameSession`
- 把 HTTP 请求翻译成 `WebGameSession` 调用

它不做的事有：

- 不直接操作 `Judger`
- 不直接操作 `Body`
- 不直接处理回合规则

换句话说，`LocalHost` 是“HTTP 适配器”，不是“游戏控制器”。

### 5.2 API 一览

#### `GET /api/strategies`

作用：

- 返回可选模式列表

来源：

- `WebGameSession.GetStrategies()`

约定：

- `1` 固定表示 `Manual`
- 从 `2` 开始依次映射到内置 AI

#### `POST /api/start`

输入：

- `mode`

作用：

- 新建一局游戏
- 重置当前模式
- 根据模式决定是否绑定 AI

返回：

- `ok`
- `snapshot`

#### `POST /api/declare`

输入：

- `skillName`
- `param`
- `esn`
- `ep`

语义：

- `skillName` / `param` 是玩家本回合声明
- `esn` / `ep` 是敌方手动模式下的声明
- 若当前不是手动模式，则敌方声明会被 AI 输出覆盖

返回：

- `ok`
- `message`
- `snapshot`

#### `GET /api/status`

作用：

- 获取当前最新快照

适用场景：

- 前端初始化
- 操作失败后重拉状态
- 调试会话层状态

### 5.3 会话对象 `WebGameSession`

`WebGameSession` 现已从 `BlacksmithCore` 移至 `BlacksmithClient/Frontend/`，是 Web 层最关键的门面对象，内部维护：

- `_strategies`
- `_game`（`GameInstance?`）
- `_activeAI`
- `_mode`
- `_started`
- `_modeName`
- `_isManual`

它的职责可以概括为四件事：

1. 管模式
   决定当前是否手动、模式名是什么、是否需要敌方 AI。
2. 管生命周期
   通过 `BackendStarter` 创建新的 `GameInstance`。
3. 管输入校验
   在真正进入引擎前先做空值、模式、技能声明合法性检查。
4. 管快照输出
   直接从领域对象的 `GetView()` 方法获取数据，组装为匿名对象返回给前端。

与旧版的关键区别在于：

- 不再依赖独立的 `WebGameReader` / `WebModels` 类
- 快照构造直接调用 `Body.GetView()` 返回的 `BodyView`，再映射到前端所需的 JSON 形状
- 回合历史直接读取 `GameHistory.SkillHistory`，而非单独的 `Turns` 列表
- `futureAttacks` 和 `futureDefenses` 作为新的快照字段，允许前端展示延迟攻击和延迟防御

### 5.4 `BackendStarter` 的存在意义

`BackendStarter` 现在非常简单，`StartBackend()` 只是返回一个新的 `GameInstance`。

虽然它目前像一层薄包装，但从设计上它提供了一个很清晰的装配点：

- 以后如果初始化一局游戏需要更多参数
- 或者想切换不同的后端构造方式
- 或者要在开局时预装某些模式

都可以优先在这里接入，而不必把逻辑直接堆到 Web 宿主里。

## 6. 一局游戏的运行时对象

### 6.1 `GameInstance`

`GameInstance` 是”一局游戏”的核心运行时容器，包含四个重要字段：

- `Player`
- `Enemy`
- `Judger`
- `History`

职责：

- 持有双方 `ActorSet`
- 提供声明合法性校验入口（`TryDeclare` / `ETryDeclare`）
- 组织主动/被动技能的收集
- 调用 `Judger` 执行本回合
- 记录历史回合声明

对外方法：

- `TryDeclare`
- `ETryDeclare`
- `Declare`
- `DeepCopy`
- `Swap`

其中 `DeepCopy()` 值得特别注意。

它不是典型的字段级深拷贝，而是：

1. 创建一个全新的 `GameInstance`
2. 重新遍历 `History.SkillHistory`
3. 逐回合调用 `Declare(...)` 重放历史

这说明：

- AI 模块的模拟局面复制是”重放式复制”
- 复制结果依赖历史声明能确定性地重建状态
- 如果后续加入强随机行为，这个设计需要额外处理随机源

### 6.2 `GameHistory`

`GameHistory` 只保留核心的 `SkillHistory`：

- `SkillHistory`：`List<(ISkillContext, ISkillContext)>`，保存每一回合双方的原始声明上下文

回合记录在旧版中曾分为 `SkillHistory`（内部重放用）和 `Turns`（前端展示用）两类。当前已将展示职责上移到 `WebGameSession`，后者在构造快照时直接从 `SkillHistory` 中提取所需信息。

`Swap()` 会把双方历史对调，说明它服务于从另一方视角重组历史的场景。

### 6.3 `ActorSet`

`ActorSet` 表示一方阵营，当前结构较简单：

- `Focus`
- `ActorList`

默认情况下：

- `Focus` 是当前主要角色
- `ActorList` 初始只有 `Focus`

这意味着当前引擎虽然以“一方一个主角色”为主，但已经预留了“阵营下多个 `Body`”的结构基础。

### 6.4 `Body` 与 `BodyView`

`Body` 是最核心的领域聚合对象之一，聚合了：

- `Skill`
- `Health`
- `Defense`
- `Resource`
- `Effect`
- `TurnContext`
- `Community`

可以把它理解成”角色当前状态 + 本回合缓存 + 所属阵营引用”。

很多逻辑最终都落在 `Body` 上，例如：

- 资源消耗与获得
- 防御叠加与更新
- 效果执行与清理
- 回合内 `Resolution` 写入

`Body.GetView()` 方法会返回一个 `BodyView` 记录，包含：

- `ProfessionNames`：从 `Skill.GetView()` 获取的职业名列表
- `HP` / `MHP`：当前生命和最大生命
- `DefenseView`：来自 `Defense.GetView()`，每个防御的名称和强度
- `ResourcesView`：来自 `Resource.GetView()`，每种资源的名称和当前数量
- `FutureAttackView`：来自 `TurnContext.GetFutureAttackView()`，延迟攻击的名称、延迟回合和力度
- `FutureDefenseView`：来自 `TurnContext.GetFutureDefenseView()`，延迟防御的名称、延迟回合和力度

`BodyView` 是前端快照的数据来源。Web 层不再通过独立的 `WebGameReader` 类读取内部对象，而是由各领域对象自行提供 `GetView()` 方法。这样做的好处是：

- 视图逻辑与领域逻辑放在同一文件中，减少”修改字段、忘记同步视图”的问题
- 新增资源/防御类型时，`GetView()` 自动反映变化，前端不需额外修改映射代码

### 6.5 `Health`

`Health` 管理：

- `HP`
- `MHP`

职责：

- 增减当前生命
- 增减最大生命
- 保证 `GainHP` 不超过 `MHP`

### 6.6 `Resource`

`Resource` 负责资源系统，内部通过 `ResourceTemplate` 建立普通资源与 Gold 资源的配对关系。

关键点：

- 构造时会反射 `ResourceType` 的枚举字典
- 名字形如 `Gold_Xxx` 的资源会自动与 `Xxx` 配对
- `Check` / `Use` 默认会同时考虑 Common 与 Gold
- 可以通过 `ifCommonOnly` 限制只使用普通资源

当前 `Resource.GetView()` 会遍历 `_resources` 字典中的所有键，输出全部已注册资源的名称和数量。

这说明：

- 引擎层支持资源扩展
- `GetView()` 自动包含所有资源类型，包括 Mod 扩展的新资源
- 前端不再需要手动维护资源映射表

### 6.7 `Defense`

`Defense` 是 `DefenseBase` 的容器，负责：

- 维护防御列表
- 按 `DefenseType` 排序
- 合并允许合并的同类防御
- 在每回合 `Update()` 中清理失效防御

防御生效顺序依赖 `DefenseType.BEValue` 的优先级。

这非常重要，因为：

- `AttackResolution` 在处理防御时会依次遍历当前防御列表
- 不同防御类型的优先级会直接改变伤害结算结果

`GetView()` 方法返回 `List<(string name, int power)>`，直接输出当前所有防御的名称和强度，供前端展示使用。

### 6.8 `Effect`

`Effect` 持有多个 `EffectEntity`，核心能力是：

- `Add`
- `AddRange`
- `Execute`
- `Update`

其中 `Execute` 会：

- 先按 `EffectType` 过滤
- 处理 `DelayTimes`
- 处理 `RemainingTimes`
- 触发 `EffectEntity.Execute`

### 6.9 `TurnContext`

`TurnContext` 是当前回合最关键的暂存结构之一，保存四类待执行分支：

- `AttackResolutions`
- `DefenseResolutions`
- `ResourceResolutions`
- `EffectResolutions`

它不是永久状态，而是”本回合内待结算的行为缓存”。

在 DSL 与判定引擎之间，`TurnContext` 起到桥梁作用：

- DSL 编译得到的 `Intent`
- `Intent.Execute` 把各种 `Resolution` 写入 `TurnContext`
- `JudgeRuleManager` 在不同阶段读取并消费这些 `Resolution`

此外，`TurnContext` 提供了两个视图方法，用于前端展示延迟中的攻击和防御：

- `GetFutureAttackView()`：返回 `List<(string name, int delayRounds, int power)>`，列出所有尚未到时的攻击分支
- `GetFutureDefenseView()`：返回 `List<(string name, int delayRounds, int power)>`，列出所有尚未到时的防御分支

这两个方法让前端可以直观展示”下一回合即将发生的效果”，提升用户体验。

## 7. 技能系统

### 7.1 `Skill` 聚合器

`Skill` 类是角色可用技能的统一入口，内部维护一个 `Package` 列表。

每个 `Package` 包含：

- `Name`
- `SkillPackage`
- `IsActive`

默认情况下，`Skill` 会先装入一个 `Common` 包。

这意味着：

- 所有人物开局至少拥有 `Common`
- 职业系统本质上是在 `Common` 的基础上继续叠加技能包

`Skill` 对外主要提供：

- `AddPackage`
- `AddSkill`
- `RemoveSkill`
- `TryDeclare`
- `Declare`
- `GetPassiveSkill`
- `GetAvailableSkillNames`
- `GetActivePackageNames`
- `GetView()`：返回除 `Common` 以外的职业包名称列表，用于前端展示职业信息
- `HaveProfession`：判断是否已转职（`_packages.Count > 1`）

### 7.2 技能声明语义

`TryDeclare(skillName, sc)` 的返回值有三种：

- `Success`
- `Illegal`
- `Rejected`

含义区别：

- `Illegal`
  当前没有任何激活中的技能包提供这个技能名。
- `Rejected`
  技能名存在，但对应的 `XxxCheck` 条件不成立。
- `Success`
  技能名存在且声明条件成立。

这个区分在前端和调试时很有用，因为它能帮助判断问题到底出在：

- 技能根本未注册
- 技能已注册但资源或前置条件不足

### 7.3 `SkillPackageBase`

`SkillPackageBase` 是技能包系统的核心基类。

它在构造时会通过反射扫描当前类的私有实例方法，并自动配对：

- `private bool XxxCheck(ISkillContext sc)`
- `private DSL.SourceFile Xxx(ISkillContext sc)`

配对规则：

- 方法名以 `Check` 结尾的布尔方法视为校验函数
- 去掉 `Check` 后得到技能名
- 再查找同名、返回 `DSL.SourceFile` 的生成函数
- 技能名统一转成小写

因此，一个技能的两个核心阶段是：

1. 声明合法性检查
2. 生成 DSL 脚本

这两个阶段天然由同一个类维护，避免“技能是否能用”和“技能具体怎么执行”分散在不同文件里。

### 7.4 `MainProfession` 与 `ProfessionModifier`

职业包基类分两类：

- `MainProfession`
- `ProfessionModifier`

含义：

- `MainProfession`
  真正可以被玩家持有的主职业包。
- `ProfessionModifier`
  挂在某个职业包上的补丁或扩展技能集。

典型扩展模式是：

1. 写一个新的 `MainProfession`
2. 再写一个 `[IsProfessionModifier(nameof(Common))]` 的修改器
3. 通过 `Common` 暴露转职技能入口

### 7.5 `ProfessionRegistry`

`ProfessionRegistry` 维护两类信息：

- `Professions`
  所有主职业名
- `_modifiers`
  每个目标职业对应的修改器列表

`SkillPackageBase` 在构造时如果发现自己是 `PackageType.Main`，会调用 `ProfessionRegistry.AddModeOnInit(this)`。

这个机制意味着：

- 主职业初始化时就会把“挂在自己身上的修改器技能”合并进来
- 合并方式是直接把修改器的 `AvailableSkillNames`、`SkillChecker`、`SkillSourceFileGenerator` 拷贝到主职业包实例上

这是一种很直接的“启动时装配，运行时使用”的扩展模型。

### 7.6 内置职业的设计模式

当前内置主职业位于：

- `Blacksmith/BlacksmithCore/Backend/SkillPackages/Logic/BuitinProfessions`

主要包括：

- `Common`
- `Warlock`
- `Cannon`
- `Driver`
- `BloodSigil`
- `Lancer`

以 `Common` 为例，可以看出三个重要模式：

1. 基础技能
   如 `iron`、`stick`、`shield`，负责资源积累、攻击、防御。
2. 转职技能
   如 `warlock`、`cannon`、`bloodsigil`，调用 `AddPackage(new Xxx())` 完成职业切换。
3. 动态规则技能
   如 `reflect`，通过 `LinkJudgeRuleDynamic(...)` 改写部分判定阶段行为。

## 8. 技能 DSL

### 8.1 为什么需要 DSL

这个项目没有让技能函数直接修改战场最终状态，而是先把技能翻译为一个中间表示 `DSL.SourceFile`。

这层中间表示带来的收益是：

- 技能定义与判定流程解耦
- 技能可以先“写意图”，再由统一判定链消费
- 同一技能可以组合攻击、防御、资源、效果、自由动作
- 动态规则可以在编译阶段挂入，而不是到处散落在业务代码里

### 8.2 `SourceFile` 的本质

`DSLforSkillLogic.SourceFile` 本质上是一个脚本构造器，内部维护：

- `_sentences`
- `_rhetoricCache`
- `_mutationsOnCompile`
- `_owner`

可以把它理解成：

- `_sentences`
  主脚本正文
- `_rhetoricCache`
  依附在某个主语句上的附加修辞，例如攻击后追加吸血效果
- `_mutationsOnCompile`
  需要在编译阶段注册给 `Judger` 的动态规则

### 8.3 DSL 支持的主要写法

#### `WriteAttack(...)`

作用：

- 往当前角色的 `TurnContext.AttackResolutions` 写入一个攻击分支

能力：

- 指定攻击力
- 指定攻击类型
- 指定 AP 因子
- 指定延迟回合

生成的 `AttackResolution.Execute` 真正落地时会：

- 遍历目标当前防御
- 根据防御类型调整攻击
- 处理命中护甲的阶段钩子
- 最终扣除目标 HP

#### `WriteDefense(...)`

作用：

- 往 `TurnContext.DefenseResolutions` 写入防御分支

真正执行时：

- 把指定 `DefenseBase` 实例加到目标 `Body.Defense` 中

#### `WriteResource(...)`

作用：

- 往 `TurnContext.ResourceResolutions` 写入资源变化分支

#### `WriteEffect(...)`

作用：

- 往 `TurnContext.EffectResolutions` 写入效果分支

它不会直接把效果实体放到角色上，而是先写 `EffectResolution`，等判定阶段再生效。

#### `WriteRecovery(...)`

作用：

- 直接给当前角色恢复生命

#### `WriteFree(...)`

作用：

- 写入任意自定义动作

这一类动作最灵活，也最需要自律。它可以直接操作：

- HP
- 资源
- 技能包
- 任意领域状态

因此它也是最容易绕开统一规则链的地方。

#### `UseResource(...)`

作用：

- 作为语法糖，通过 `WriteFree` 完成资源消耗

### 8.4 修辞式扩展

`AttackFile` 提供了一些依附在攻击上的增强能力，例如：

- `WithFree(AttackStage, action)`
- `WithBloodSuck(percent)`
- `WithInterupt()`

这些不是独立攻击，而是附着在“刚刚写入的那次攻击”上的阶段钩子。

因此 DSL 支持两层表达：

- 主语句：真正写入攻击、防御、资源、效果
- 修辞语句：修改主语句在某个阶段的表现

### 8.5 编译结果 `Intent`

`SourceFile.Compile(judger)` 会做两件关键事情：

1. 若传入 `judger`，把 `_mutationsOnCompile` 中的动态规则注册到 `JudgeRuleManager`
2. 把所有句子合并成一个 `Intent.Execute`

`Intent` 很轻，只包含一个：

- `Action<ActorSet> Execute`

它的职责不是立刻结算结果，而是“把本技能要做的事情写到当前出手方的上下文里”。

## 9. 从声明到结算：一个回合的完整链路

下面按真实代码路径把一个标准回合串起来。

### 9.1 第一步：前端提交声明

前端调用：

- `POST /api/declare`

`LocalHost` 解析 DTO 后转给：

- `WebGameSession.DeclareTurn(...)`

### 9.2 第二步：会话层校验

`WebGameSession.DeclareTurn` 会依次做：

1. 检查游戏是否已经开始
2. 检查对局是否已经结束
3. 检查玩家输入是否为空
4. 调用 `gameInstance.TryDeclare(...)` 校验玩家技能
5. 根据模式决定敌方技能来源
6. 若为 AI 模式，则调用 `selectedAI.ChooseSkill(...)`
7. 再次校验敌方技能

直到这里，回合还没有真正写入引擎状态。

### 9.3 第三步：`GameInstance.Declare(...)`

一旦校验通过，真正执行：

1. 为双方创建 `DefaultSkillContext`
2. 把本回合声明写入 `History.SkillHistory`
3. 玩家主动技能生成一个 `SourceFile`
4. 玩家被动技能列表插入到主动技能前面
5. 敌方主动技能生成一个 `SourceFile`
6. 敌方被动技能列表插入到主动技能前面
7. 调用 `Judger.Judge(playerSfs, enemySfs)`
8. 读取结算结果并写入 `History.Turns`

这里有一个非常重要的设计：

- 每方不是只有一个 `SourceFile`
- 而是“一个主动技能 + 多个被动技能 `SourceFile`”

这意味着：

- 被动技能在架构上和主动技能一样，都是 DSL 脚本
- 差别只在于它们由 `GetPassiveSkill` 自动产出

### 9.4 第四步：`Judger` 编译 SourceFile

`Judger.Judge` 内部会：

1. 对玩家 `SourceFile` 列表执行 `Compile`
2. 对敌方 `SourceFile` 列表执行 `Compile`
3. 把得到的 `Intent` 分别存入 `_playerIntents` 和 `_enemyIntents`
4. 执行 `_onJudge`

其中 `Compile(List<SourceFile>)` 会对列表中的第一个 `SourceFile` 做一次特殊处理：

- 先取出 `sourceFiles[0]`
- 编译其余项
- 再把这个第一个 `SourceFile` 对应的 `Intent` 插回结果最前面

结合 `GameInstance.Declare(...)` 的组装方式看，这个“第一个项”通常会是第一个被动技能脚本，而不是单独的聚合对象。

### 9.5 第五步：`Intent` 写入 `TurnContext`

`Judger.TranslateIntentsToResolutions()` 会：

- 先执行所有玩家 `Intent.Execute(Player)`
- 再执行所有敌方 `Intent.Execute(Enemy)`

执行结果通常不是立刻修改最终结局，而是：

- 向当前出手方 `Focus.TurnContext` 写入攻击、防御、效果、资源等 `Resolution`
- 或者在个别需要即时修改状态的场景下直接执行自由动作

### 9.6 第六步：按判定阶段执行规则链

`JudgeRuleManager.GetRule()(Player, Enemy)` 会把多个阶段串成一条大的规则链，依次执行：

1. `OnBegin`
2. `OnEffectTaking_AfterResolutionWritten`
3. `OnEffectSwaping`
4. `OnAttackCanceling`
5. `OnAttackSwaping`
6. `OnApplyingEffect`
7. `OnEffectTaking_AfterTransport`
8. `OnApplyingOthers`
9. `OnUpdating`
10. `OnEffectTaking_AfterResult`
11. `OnEnd`

### 9.7 第七步：写回前端快照

回合结束后：

- `WebGameSession` 调用 `BuildSnapshot()`
- 通过 `Body.GetView()` 获取双方 `BodyView`
- 直接从 `GameHistory.SkillHistory` 提取回合记录
- 组装为匿名对象，通过 API 返回给前端

## 10. 判定引擎

### 10.1 `Judger`

`Judger` 是回合级执行器，持有：

- `Player`
- `Enemy`
- `JudgeRuleManager`

它的职责不是定义规则，而是做三件事：

1. 编译双方技能脚本
2. 把编译结果写入双方 `TurnContext`
3. 触发规则链执行

可以把 `Judger` 理解成“回合调度器”。

### 10.2 `JudgeRuleManager`

`JudgeRuleManager` 才是真正组织“回合阶段”的地方。

内部核心结构是：

- `SortedDictionary<JudgeStage.BEValue, StageRuleContainer>`

每个 `StageRuleContainer` 包含：

- 一个基础规则 `_baseRule`
- 多个覆盖规则 `_overrideRules`
- 多个前置修饰 `_modifiersBefore`
- 多个后置修饰 `_modifiersAfter`

这使每个阶段都支持：

- 默认行为
- 前后插桩
- 整体替换

### 10.3 默认阶段规则含义

#### `OnBegin`

默认空操作。

适合作为：

- 每回合前置初始化
- 某些动态规则的起始挂钩

#### `OnEffectTaking_AfterResolutionWritten`

执行 `EffectType.AfterResolutionWritten` 对应的效果实体。

适合：

- 在回合意图写入后立刻触发的效果

#### `OnEffectSwaping`

处理 `EffectResolution` 的目标交换逻辑。

默认实现会把指向敌方或立即生效的效果在双方列表间交换。

#### `OnAttackCanceling`

处理双方攻击对消。

默认逻辑会：

- 跳过真实伤害
- 跳过延迟攻击
- 对符合条件的攻击按类型优先级进行对消

#### `OnAttackSwaping`

处理攻击目标交换。

默认实现会交换双方当回合生效的攻击分支。

#### `OnApplyingEffect`

执行 `TurnContext.ExecuteEffect(...)`，让当回合到时的 `EffectResolution` 真正落到目标身上。

#### `OnEffectTaking_AfterTransport`

执行 `EffectType.AfterTransport` 对应的效果实体。

#### `OnApplyingOthers`

按顺序执行：

1. 防御
2. 攻击
3. 资源

这一顺序很关键，因为它定义了“本回合的生效次序”。

#### `OnUpdating`

调用双方所有 `Body.Update()`，用于：

- 防御状态清理
- 效果状态清理

#### `OnEffectTaking_AfterResult`

执行 `EffectType.AfterResult` 对应的效果实体。

#### `OnEnd`

默认空操作。

适合作为：

- 回合收尾逻辑
- 下回合前的最终整理

### 10.4 `StageRuleContainer` 的生存期模型

每条动态注入的规则被包装为 `RuleUnit`，包含：

- `RemainingRounds`
- `DelayRounds`
- `Rule`

`Build()` 前会先 `Update()`：

- 清掉过期规则
- 推进剩余回合和延迟回合

因此动态规则天然支持：

- 延迟生效
- 生效若干回合
- 在特定阶段之前或之后插入

## 11. 动态规则系统

### 11.1 为什么需要动态规则

普通 DSL 很适合描述：

- 造成伤害
- 增加防御
- 消耗资源
- 增加效果

但不擅长描述“改写判定流程本身”的技能，例如：

- 反射
- 转移
- 延迟保护
- 某阶段覆盖

因此项目额外提供了动态规则系统。

### 11.2 参与的核心类型

- `DynamicJudgeRuleName`
- `Mutation`
- `JudgeStage`
- `DynamicJudgeRulePool`
- `JudgeRuleManager`

### 11.3 工作机制

一条动态规则从技能到生效的路径如下：

1. 技能在 DSL 中调用 `LinkJudgeRuleDynamic(ruleKey, mutations)`
2. `SourceFile.Compile(judger)` 时把这些 `mutations` 注册到 `judger.JudgeRuleManager`
3. `JudgeRuleManager.RegistJudgeRuleDynamic(...)` 交给 `_dynamicPool`
4. `JudgeRuleManager.AddJudgeRule(source, name)` 根据施法者把规则专门化
5. 专门化后的规则插入对应 `JudgeStage` 的 `StageRuleContainer`
6. 当本回合执行到该阶段时，动态规则参与执行

### 11.4 为什么需要“按施法者专门化”

默认规则通常是对称的：

- 玩家和敌人的处理逻辑一样

但像 `reflect` 这种技能不是纯对称规则，它必须知道：

- 究竟是谁触发了这条动态规则

所以 `DynamicJudgeRulePool` 内部存的不是最终 `Mutation`，而是 `MutationPrototype`。

当 `Query(source, name)` 时，原型会根据 `source` 生成真正的 `Mutation`，把：

- 施法者视角
- player/enemy 当前参数顺序

统一折叠进最终规则。

### 11.5 `reflect` 是如何落地的

以 `Common.Reflect` 为例，它通过：

- `DynamicJudgeRuleName.Instance.Reflect()`
- 在 `OnEffectSwaping` 和 `OnAttackSwaping` 阶段插入 `After Modifier`

来改变默认阶段行为。

这说明动态规则系统的定位不是“另起一套战斗引擎”，而是：

- 在已有默认阶段链上做局部改写
- 尽量复用既有判定流程

## 12. 防御、攻击与效果的结算细节

### 12.1 攻击如何落地

`WriteAttack` 最终生成 `AttackResolution`，执行时会：

1. 读取目标 `Defense` 列表
2. 若不是真实伤害，则按顺序遍历防御
3. 根据不同防御类型执行 `Work(...)`
4. 触发攻击阶段钩子
5. 将剩余伤害打到目标 `Health`

这一过程里，`DefenseType` 的优先级顺序会直接影响伤害链。

### 12.2 防御对象不是纯数值

每个防御都是一个 `DefenseBase` 子类对象，而不是简单数字。

因此它们可以各自封装：

- 是否可合并
- 如何更新寿命
- 如何处理攻击
- 是否带副作用

例如：

- `CommonArmor`
  吸收伤害并消耗自身耐久
- `CommonReduction`
  提供一次性减伤
- `ThornReduction`
  在吸收物理伤害时追加延迟反击

### 12.3 效果系统分两层

效果相关对象至少有两层：

- `EffectResolution`
  本回合待施加的效果意图
- `EffectEntity`
  已经挂在角色身上的持续效果实体

这层分离的好处是：

- 可以先参与本回合规则链
- 再在合适的阶段把效果真正附着到角色
- 让“效果施加时机”和“效果持续执行逻辑”分开

## 13. 前端快照模型

### 13.1 从 `WebGameReader` 到分布式 `GetView()`

在旧版架构中，快照构造由一个独立的 `WebGameReader` 类负责，它读取引擎内部对象并手动映射到 `WebGameSnapshot` / `WebActorSnapshot`。

当前架构已将这一职责拆散到各领域对象中：

- `Body.GetView()` → `BodyView`
- `Defense.GetView()` → `List<(string name, int power)>`
- `Resource.GetView()` → `List<(string name, float quantity)>`
- `TurnContext.GetFutureAttackView()` → `List<(string name, int delayRounds, int power)>`
- `TurnContext.GetFutureDefenseView()` → `List<(string name, int delayRounds, int power)>`
- `Skill.GetView()` → `List<string>`（职业名列表）

`WebGameSession.BuildSnapshot()` 直接调用这些方法，组装为匿名对象返回给前端。

这种分布式设计的好处是：

- 新增资源/防御类型时，`GetView()` 自动反映变化
- 视图逻辑与领域逻辑在同一文件中，修改字段时不易遗漏视图更新
- 不再需要一个独立的”阅读器”类去理解所有领域对象的内部结构

### 13.2 快照的 JSON 结构

当前 API 返回的快照包含：

- `started`：游戏是否已开始
- `modeName`：当前模式名称
- `manualMode`：是否为手动模式
- `result`：当前结果（`”win”` / `”lose”` / `”draw”` / `”next”`）
- `player` / `enemy`：双方角色视图，包含：
  - `professions`：已激活职业包名称列表
  - `hp` / `maxHP`：当前生命与最大生命
  - `defenses`：防御列表（`name`、`power`）
  - `resources`：资源列表（`name`、`quantity`）
  - `futureAttacks`：延迟攻击列表（`name`、`delayRounds`、`power`）
  - `futureDefenses`：延迟防御列表（`name`、`delayRounds`、`power`）
  - `availableSkills`：当前可声明技能名列表
- `turns`：回合历史列表，每条包含 `index`、`result`、`playerSkill`、`playerParam`、`enemySkill`、`enemyParam`

### 13.3 可声明技能列表的生成逻辑

`Skill.GetAvailableSkillNames()` 直接返回所有激活包中的所有可用技能名。

与旧版不同，当前实现不再对每个技能做参数试探过滤，而是直接列出全部已注册的技能名。这意味着：

- 前端展示的技能列表与引擎注册的技能名完全一致
- 技能是否真正可用（资源是否足够等）由 `TryDeclare` 在声明时校验
- 校验失败时会返回对应的 `SkillDeclareResult`（`Illegal` 或 `Rejected`）

### 13.4 前端类型校验

`wwwroot/js/types.js` 定义了一套轻量的运行时类型校验，对 API 返回的快照进行形状检查：

- `SnapshotShape`：校验顶层快照字段的存在性和类型
- `ActorShape`：校验角色子对象的字段

这些校验在开发阶段帮助快速发现前后端格式不一致的问题，但不影响生产环境的正常功能（校验失败仅输出 `console.warn`）。

### 13.5 当前快照的扩展性

由于 `GetView()` 方法遍历的是内部完整数据结构，新增资源、防御类型、延迟攻击或延迟防御都会自动反映在快照中，前端不需额外修改映射代码。

但以下部分仍需要额外配合：

- 前端技能说明文本
- 特殊的 UI 渲染逻辑
- 更复杂的序列化协议

也就是说，引擎层扩展成功后，UI 层基本数据展示可以自动感知，但定制化的展示逻辑仍需手动更新。

## 14. AI 模块

### 14.1 接口层

AI 统一实现 `IAIStrategy`：

- `string Name`
- `void Init(GameInstance gameInstance)`
- `(string skillName, int param) ChooseSkill(ActorSet self, ActorSet opponent)`

这说明 AI 的接入位置在会话层之上、规则层之下：

- 它不负责启动会话
- 也不直接改写判定引擎
- 它只是替敌方提供本回合声明

### 14.2 `BloodSigilStrategy`

这是一个偏规则型、启发式的 AI。

特点：

- 前几回合固定积攒资源
- 指定回合转 `BloodSigil`
- 再按对手职业与当前血量做分支决策

适合：

- 作为具体职业的样板 AI
- 展示如何基于当前状态做硬编码策略

### 14.3 `GeneralStrategy`

这是一个更通用的搜索型 AI。

核心特征：

- 持有 `GameInstance _main`
- 通过 `DeepCopy()` 复制局面
- 用多线程运行近似 MCTS
- 通过参数化评分函数评估局面

它的依赖点很清晰：

- `GameInstance.DeepCopy()`
- `TryDeclare` / `ETryDeclare`
- `Declare`
- `History.SkillHistory.Count`
- 当前双方 `Body` 的 HP、资源、职业状态

换句话说，`GeneralStrategy` 说明当前引擎已经支持：

- 在纯内存里重放对局
- 通过重复模拟来做 AI 搜索

### 14.4 AI 与宿主的关系

宿主层不会深入关心 AI 的算法细节，只做三件事：

1. 启动时收集可用 AI
2. 开局时绑定当前选择的 AI
3. 敌方出手时调用 `ChooseSkill`

这种解耦是合理的：AI 是“决策模块”，不是“规则模块”。

## 15. 插件与扩展机制

### 15.1 扩展入口的本质

当前扩展体系的核心思想是：

- 启动时扫描 DLL
- 反射发现类型或静态方法
- 把扩展装进统一注册表
- 运行期直接使用装配后的结果

因此这是典型的“启动期装配型插件架构”，而不是“运行中热插拔型插件架构”。

### 15.2 可扩展枚举 `BlacksmithEnum<T>`

`BlacksmithEnum<T>` 的设计非常关键。

它提供的不是普通 `enum`，而是：

- 运行时可创建成员
- 成员带优先级
- 成员有稳定可比较值
- 成员可以通过方法名映射到可读名称

因此像下面这些概念都可以被扩展：

- `ResourceType`
- `DefenseType`
- `AttackType`
- `EffectType`
- `EffectTargetType`
- `DynamicJudgeRuleName`
- `JudgeStage`

扩展带来的最大收益是：

- Mod 不只是加新技能，还能加新“规则分类”

### 15.3 枚举扩展方法的约定

带 `[IsBlacksmithEnumModifier]` 的静态类中，可以写：

- 带 `[IsBlacksmithEnumMember(priority)]`
- 只有一个参数
- 参数类型为目标枚举类型
- 返回值为该枚举对应 `BEValue`

的方法来新增枚举成员。

例如：

- `public static ResourceType.BEValue Cross(this ResourceType resourceType) => ResourceType.GetBEValue();`

### 15.4 职业扩展的约定

新增职业通常包含两部分：

1. `MainProfession`
   真正定义职业技能
2. `ProfessionModifier`
   挂到 `Common` 上暴露转职入口

如果只写主职业，不写 `Common` 修改器，那么理论上职业名会被注册，但玩家默认未必有可到达这个职业的入口技能。

### 15.5 当前扩展体系的边界

当前插件机制可以很好扩展：

- 资源
- 防御类型
- 判定阶段
- 动态规则名
- 职业包
- 技能实现

但以下部分仍需要额外配合：

- 前端资源显示
- 前端技能说明
- 更复杂的序列化协议

也就是说，引擎层扩展成功，不代表 UI 层自动感知。

## 16. 典型扩展路径

### 16.1 新增一个普通职业

建议路径：

1. 新建一个继承 `MainProfession` 的类
2. 用 `XxxCheck` / `Xxx` 方式定义职业技能
3. 新建一个继承 `ProfessionModifier` 的类
4. 给修改器加 `[IsProfessionModifier(nameof(Common))]`
5. 在修改器里新增一个转职技能，内部 `AddPackage(new YourProfession())`
6. 将编译出的 DLL 放到宿主运行目录，让启动扫描到

### 16.2 新增一种资源

建议路径：

1. 在插件程序集里写一个带 `[IsBlacksmithEnumModifier]` 的静态类
2. 为 `ResourceType` 增加新成员
3. 若有 Gold 配对资源，命名遵循 `Gold_Xxx`
4. 在技能 DSL 中通过 `WriteResource` / `UseResource` 使用它
5. 前端展示由 `Resource.GetView()` 自动处理，无需额外修改快照层

### 16.3 新增一种改变规则阶段的技能

建议路径：

1. 先判断是否可以只靠普通 DSL 实现
2. 如果不行，再定义 `DynamicJudgeRuleName`
3. 为技能准备 `Mutation` 列表
4. 在技能 `SourceFile` 上调用 `LinkJudgeRuleDynamic`
5. 让规则挂到合适的 `JudgeStage`
6. 明确它应当是 `Override`、`Before Modifier` 还是 `After Modifier`

## 17. 维护与排障指南

下面按照最常见的问题类型给出推荐排查路径。

### 17.1 前端显示不对

先看：

- `BlacksmithClient/wwwroot/js/*`
- `Body.GetView()` 及相关 `GetView()` 方法
- `WebGameSession.BuildSnapshot()`

重点检查：

- 快照字段名是否和前端假设一致
- 新增资源/技能/职业的 `GetView()` 是否返回了预期数据
- `types.js` 是否对快照形状报出警告

### 17.2 技能名存在但按钮不可用

先看：

- `Skill.GetAvailableSkillNames()`
- `TryDeclare`
- 对应 `XxxCheck`

重点区分：

- 是技能未注册（不会出现在 `AvailableSkillNames` 中）
- 还是技能注册了但 `XxxCheck` 校验不通过（返回 `Rejected`）

### 17.3 技能声明成功但结算结果不对

先看：

- 对应技能生成的 DSL
- `Intent` 是否正确写入 `TurnContext`
- 当前 `JudgeStage` 顺序是否符合预期
- `Defense` / `Effect` / `Resource` 的执行时机是否正确

可按这个顺序检查：

1. `SkillPackageBase` 是否找到了正确的生成方法
2. `Declare` 得到的 `SourceFile` 是否合理
3. `Compile` 时是否注册了动态规则
4. `TurnContext` 中是否存在预期的 `Resolution`
5. `JudgeRuleManager` 是否在正确阶段消费了这些 `Resolution`

### 17.4 回合流程整体异常

先看：

- `Judger`
- `JudgeRuleManager`
- `DynamicJudgeRulePool`
- `JudgeStage`

重点怀疑：

- 阶段顺序
- 动态规则覆盖顺序
- `RemainingRounds` / `DelayRounds` 处理
- 攻击和效果交换时机

### 17.5 Mod 没生效

先看：

- DLL 是否确实出现在宿主运行目录
- `PluginLoader.Initialize` 是否扫描到了该 DLL
- 类型是否满足反射约定
- 特性是否加对
- 目标职业名是否匹配

再看：

- 扩展枚举是否在 `CloseFactory` 前完成注册
- 主职业是否被注册进 `ProfessionRegistry.Professions`
- 修改器是否挂到了正确职业名

### 17.6 AI 行为异常

先看：

- `WebGameSession` 是否正确调用了 `selectedAI.Init(...)`
- `ChooseSkill` 输出的技能是否通过 `ETryDeclare`
- `GameInstance.DeepCopy()` 重放状态是否准确

特别是 `GeneralStrategy`，如果发现模拟局面和真实局面表现不一致，应优先检查：

- 历史重放是否遗漏了任何即时副作用
- 某些状态是否没有被 `Declare` 历史完全重建

## 18. 当前架构的优点与约束

### 18.1 优点

- 规则引擎与 Web 展示层隔离得比较清楚
- 技能实现方式统一，扩展成本低
- DSL + `TurnContext` + 阶段链 的组合非常适合做规则型游戏
- 动态规则系统让“改写流程”成为一等扩展能力
- AI 可以直接重放游戏状态，工程上很省心
- 插件装配逻辑集中，启动链易于理解

### 18.2 约束

- `DeepCopy()` 基于历史重放，对非确定性逻辑比较敏感
- `WriteFree(...)` 很灵活，但也容易绕过统一规则边界
- 扩展机制主要是启动期装配，不是热插拔
- `ModExamples` 更偏示例和历史参考，接入时需要以当前核心接口为准
- 快照层改为分布式 `GetView()` 后，新增资源/防御可自动展示，但定制化 UI 逻辑仍需手动更新

---

## 19. 一句话总结

Blacksmith 当前最核心的架构思想是：

把“技能描述”“回合上下文”“阶段化判定”“动态规则改写”“本地 Web 会话”拆成独立层，再由 `BlacksmithClient` 在启动时把这些层装配成一个可运行、可调试、可扩展的本地对战宿主。

如果只记住一条主线，可以记成下面这句：

`SkillPackage -> DSL.SourceFile -> Intent -> TurnContext -> JudgeRuleManager -> Body.GetView()`

这条链就是当前项目从“玩家声明一个技能”到“前端看到回合结果”的主干路径。
