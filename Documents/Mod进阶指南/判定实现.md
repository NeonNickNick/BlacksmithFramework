# 判定实现
[返回](./引言.md)

本篇介绍 Blacksmith 的判定层是如何工作的，以及动态规则为什么能在回合流程中插入、覆盖或修饰默认逻辑。

## 总体结构

一回合判定的核心由以下几层组成：

1. 技能通过 DSL 产出 `SourceFile`
2. `SourceFile.Compile()` 生成 `Intent`
3. `Judger` 收集双方 `Intent`
4. `JudgeRuleManager` 生成本回合判定规则链
5. 各阶段按固定顺序执行

相关代码主要位于：

- `Blacksmith/BlacksmithCore/Backend/JudgementLogic/Judgement/JudgeRuleManager.cs`
- `Blacksmith/BlacksmithCore/Backend/JudgementLogic/Judgement/DynamicJudgeRulePool.cs`
- `Blacksmith/BlacksmithCore/Backend/JudgementLogic/Judgement/Core/DynamicJudgeRuleName.cs`

## `JudgeStage`

当前项目里的 `JudgeStage` 不是原生 `enum`，而是一个可扩展枚举：

```csharp
public class JudgeStage : BlacksmithEnum<JudgeStage>
{
    [IsBlacksmithEnumMember(0)]
    public BEValue OnBegin() => GetBEValue();

    [IsBlacksmithEnumMember(8)]
    public BEValue OnEffectTaking_AfterResolutionWritten() => GetBEValue();

    [IsBlacksmithEnumMember(16)]
    public BEValue OnEffectSwaping() => GetBEValue();

    [IsBlacksmithEnumMember(32)]
    public BEValue OnAttackCanceling() => GetBEValue();

    [IsBlacksmithEnumMember(64)]
    public BEValue OnAttackSwaping() => GetBEValue();

    [IsBlacksmithEnumMember(128)]
    public BEValue OnApplyingEffect() => GetBEValue();

    [IsBlacksmithEnumMember(256)]
    public BEValue OnEffectTaking_AfterTransport() => GetBEValue();

    [IsBlacksmithEnumMember(512)]
    public BEValue OnApplyingOthers() => GetBEValue();

    [IsBlacksmithEnumMember(1024)]
    public BEValue OnUpdating() => GetBEValue();

    public BEValue OnEffectTaking_AfterResult() => GetBEValue();

    [IsBlacksmithEnumMember(4096)]
    public BEValue OnEnd() => GetBEValue();
}
```

这些阶段会按 `priority` 排序依次执行，因此它本身也可以被 Mod 扩展。

## `JudgeRuleManager` 在做什么

`JudgeRuleManager` 负责管理每个阶段的规则容器。每个阶段都有一个 `StageRuleContainer`，里面分成三类规则：

- `Override`
- `Modifier Before`
- `Modifier After`

每次执行一个阶段时，顺序是：

1. 先执行全部 `Before` 修饰器
2. 再执行该阶段的核心规则
   如果存在 `Override`，则使用最后一个覆盖规则
   否则使用默认基础规则
3. 最后执行全部 `After` 修饰器

因此你可以把系统理解为：

- 默认规则提供“原本的游戏流程”
- 动态规则负责在合适的阶段插入或替换行为

## 默认规则都做了什么

`JudgeRuleManager` 里已经定义了多个阶段的默认规则，例如：

- `OnEffectSwaping`：交换效果分支
- `OnAttackCanceling`：攻击互相抵消
- `OnAttackSwaping`：交换攻击分支
- `OnApplyingEffect`：应用效果分支
- `OnApplyingOthers`：依次应用防御、攻击、资源分支
- `OnUpdating`：更新角色状态

这意味着：

- 你不需要手写整套判定系统
- 大多数复杂 Mod 只需要在某个阶段补一条规则即可

## `Mutation` 是什么

动态规则的最小单位叫 `Mutation`：

```csharp
public class Mutation
{
    public int RemainingRounds;
    public int DelayRounds;
    public JudgeStage.BEValue Stage;
    public RuleType RuleType;
    public ModifierOrder ModifierOrder;
    public Action<ActorSet, ActorSet> JudgeRule;
}
```

它描述的是：

- 在哪个阶段生效：`Stage`
- 是覆盖还是修饰：`RuleType`
- 修饰器是在前还是后：`ModifierOrder`
- 持续几回合：`RemainingRounds`
- 延迟几回合开始：`DelayRounds`
- 实际执行逻辑：`JudgeRule`

## `DelayRounds` 和 `RemainingRounds`

这是最容易误解的部分。

### 结论先说

- `DelayRounds`：控制“还要再等几回合才开始进入执行倒计时”
- `RemainingRounds`：控制“规则还能再执行多少轮”

### 代码里的实际行为

`StageRuleContainer.RuleUnit.TimePass()` 的逻辑是：

```csharp
if (DelayRounds > 0)
{
    DelayRounds--;
}
else
{
    RemainingRounds--;
}
```

`Build()` 只会选出 `RemainingRounds == 0` 的规则参与本轮执行。

因此：

- 新建一个默认 `remainingRounds = 1, delayRounds = 0` 的规则
- 第一次经过该阶段时先把 `RemainingRounds` 从 1 变成 0
- 然后该规则会在这一轮参与执行
- 下一轮再被清理掉

如果设置 `delayRounds = 1`：

- 第一轮只减少 `DelayRounds`
- 第二轮才开始减少 `RemainingRounds`
- 也就是“下一回合开始生效”

## `DynamicJudgeRulePool` 在做什么

`DynamicJudgeRulePool` 不直接保存最终规则，而是保存“规则模板”。

原因是：

- 某个动态规则通常和“是谁施放了这个技能”有关
- 只有在技能真正被某一方使用时，才能确定规则里的 `source` 应该是谁

因此它分两步：

1. `RegistDynamic(name, mutations)`
   保存模板
2. `Query(source, name)`
   针对本次施法者 `source`，把模板专门化成真正可执行的 `Mutation`

## `DynamicJudgeRuleName`

动态规则名称本身也是一个可扩展枚举：

```csharp
public class DynamicJudgeRuleName : BlacksmithEnum<DynamicJudgeRuleName>
{
    [IsBlacksmithEnumMember(0)]
    public BEValue Reflect() => GetBEValue();

    [IsBlacksmithEnumMember(1)]
    public BEValue Charge() => GetBEValue();
}
```

如果你的 Mod 需要新的动态规则键，也可以像其他扩展枚举一样去扩展它。

## 一个技能是怎么把动态规则挂进去的

当技能调用：

```csharp
.LinkJudgeRuleDynamic(ruleKey, mutations)
```

会发生两件事：

1. `SourceFile` 记住这组动态规则键值对
2. 当 `SourceFile.Compile(judger)` 时：
   调用 `judger.JudgeRuleManager.RegistJudgeRuleDynamic(ruleKey, mutations)`
   再调用 `judger.JudgeRuleManager.AddJudgeRule(_owner, ruleKey)`

也就是说：

- 先注册模板
- 再按当前施法者专门化
- 最后把专门化结果加入本回合规则容器

## 用 `Charge` 理解整个链路

`Lancer.Charge()` 是一个非常好的例子。

它做了这些事：

- 消耗资源
- 修改 `_chargeCount` 和 `_chargeCost`
- 注册一个 `OnAttackCanceling` 的 `Before Modifier`
- 注册一个 `OnBegin` 且 `delayRounds: 1` 的 `Before Modifier`

效果上等价于：

- 当回合先挂一个“如果对手有即时攻击，就自动反击”的规则
- 再挂一个“下回合开始检查是否需要重置蓄力状态”的规则

## 什么时候该用动态规则

适合：

- 改写阶段行为
- 下一回合触发的机制
- 反击、转移、延迟保护、特殊判定链
- 与默认规则强耦合的效果

不适合：

- 单纯的攻击、防御、资源变化
- 只在当前技能本身里就能完成的结算

如果一个效果只需要“写一个攻击/恢复/防御分支”，优先用普通 DSL，不要上升到动态规则。
